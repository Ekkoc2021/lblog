# Claude Code 开发实践：从需求到交付的方法论

本文通过一个后端缓存模块的真实案例，抽象出一套使用 Claude Code 进行特性开发的**通用方法论**。重点关注：人做什么、Agent 做什么、每个阶段的价值是什么。

---

## 一、核心工作模型

```
设计  →  开发  →  Agent审查  →  测试  →  修复  →  复测  →  人工验收
 │         │         │           │        │        │         │
 人        Agent     Agent       Agent    人       Agent      人
(决策)    (执行)   (模式检查)    (执行)   (决策)   (回归验证)  (审查+复测)
```

前六个阶段是一轮**人+Agent 协作闭环**：人设计、Agent 实现、Agent 审查、Agent 测试、人决策修复、Agent 复测。第七个阶段是**人作为最终验收者**，用完全不同的视角审视交付物。

---

## 二、阶段拆解：协作闭环

### 2.1 设计阶段（人）

**做什么：** 在写一行代码之前，把"要做什么"写成设计文档。本次输出了两份：架构设计（模块结构、技术选型）和失效策略（穷举所有写操作，输出 8×5 失效矩阵）。

**为什么必须人做：**
- 架构决策需要全局视角。比如"文章增删改要不要清理分类缓存"，涉及两个模块的耦合取舍，Agent 看不清全貌
- 设计文档是后续所有 Agent 的**唯一工作依据**。dev-agent 按它写代码，tester-agent 按它设计预期，没有文档就会出现各自理解偏差
- 设计越精确，Agent 产出越正确。失效矩阵精确到"哪个 Controller、哪个方法、清哪个缓存"，Agent 没有任何理解歧义

**这一阶段不可省略。** 跳过设计直接让 Agent 写代码，结果就是"看起来能跑但漏了一堆边界情况"——本次审查发现的 reportView 遗漏就是例子。

---

### 2.2 开发阶段（Agent 执行，人验证）

**做什么：** 把设计文档交给 dev-agent，让它实现全部代码。本次 13 个文件，4 个 CacheService 结构完全一致、7 个 Controller 改动模式相同——Agent 做这种事比人手写更快且不会漏。

**怎么给 Agent 下指令：**
- 指令必须精确到文件级别。不是"加个缓存"，而是"创建 CategoriesCacheService.java，用 Caffeine 10min TTL，key 是 limit，refresh() 方法是 invalidateAll()"
- 给 Agent 提供现有代码作为参考模板。本次 dev-agent 正确复用了 SiteConfigCacheService 的结构
- 要求 Agent 在完成后自己编译验证

**人验证什么：**
- 不是验证编译过没过（Agent 自己保证），而是验证**设计是否被正确翻译**——事件系统是否触发、Controller 改动是否全覆盖、有没有漏掉的写操作路径

---

### 2.3 Agent 代码审查（独立 Agent）

**做什么：** 派出一个**独立的** reviewer-agent，给它全部变更文件，让它按正确性、一致性、边界情况逐文件审查。

**为什么必须是独立 Agent：**
- dev-agent 和 reviewer 共享同一个设计文档时，会有相同的思维盲区——都假设"设计是对的，代码按设计实现就是对的"
- 独立审查只看代码，不看设计，"这代码有没有问题"的视角更纯粹
- 实际效果：本次审查发现了 8 个问题（3 个中等、5 个轻微），其中 reportView 遗漏是测试也没覆盖到的

**审查必须输出书面报告。** 口头结论不可追溯，书面报告是修复阶段的依据。

---

### 2.4 测试阶段（人设计场景，Agent 执行）

**做什么：** 人先把"什么场景该验证什么"写成测试场景文档，然后派 tester-agent 逐个执行，收集日志证据，输出测试报告。

**为什么人必须设计场景：**
- Agent 不会自己设计测试。直接说"测一下缓存模块"，Agent 只会调几个 API、看到 200 就报告通过
- 测试的价值在于**人想清楚了所有该测的路径**：首次访问该 miss、二次该 hit、写操作后该失效、不该失效的缓存必须保持 hit
- 场景设计直接来源于设计阶段的失效矩阵：矩阵里的每一项就是一个测试场景

**Agent 做什么：** 启动应用、调 API、grep 日志、截图存证、写报告。全是执行。

---

### 2.5 修复与复测（人决策，Agent 验证）

**做什么：** 审查报告出来后，人逐条判断哪些修、哪些不修、怎么修。修复后，针对修复点设计新的复测场景，派 Agent 执行。

**为什么修复必须人决策：**
- 审查发现的不是"bug"而是"问题"，修不修需要判断优先级
- 本次 8 个问题修了 3 个：reportView（功能正确性，必修）、null 安全（防御性编程）、日志格式。另外 5 个没修——双重失效（冗余但不影响正确性）、Mapper limit 参数（落实现有问题）、死代码（非紧急）
- Agent 做不了优先级判断

**为什么复测必须独立设计：**
- 复测不是重跑首轮，而是针对修复点的**新场景**
- 本次复测新增了"view 只刷新 hot_posts 不刷新 categories"的场景——首轮没设计这个，因为当时还不知道有这个问题

---

## 三、验收阶段（人）

前六个阶段构成了一个完整的 Agent 协作闭环——代码通过了开发、审查、测试、修复、复测。但所有这些阶段都有一个共同前提：**它们验证的是"做的事情对不对"，而不是"做的东西用起来好不好"。** 验收阶段由人来回答后一个问题。

验收包含两个动作：**代码审查**和**功能复测**。它们和前面 Agent 做的审查、测试不是重复，而是不同维度。

### 3.1 人工代码审查

Agent 审查和人工审查的区别：

| | Agent 审查 | 人工审查 |
|------|-----------|---------|
| 关注点 | 模式、格式、一致性、空判断 | 业务语义、设计合理性、性能隐患 |
| 方法 | 逐文件比对，看代码是否符合规范 | 逐功能走读：这段代码在干什么、为什么这样写 |
| 能发现 | 漏了 null 检查、日志格式不一致、缺少 refresh 调用 | 这个设计本身有没有问题、这个实现是否过于复杂 |
| 典型案例 | "4 个 CacheService 的日志格式不一致" | "双重失效说明缓存职责不清晰，应该统一为单一路径" |

Agent 审查本质上是**模式检查**——它比对人做得好，因为它不会遗漏任何文件、不会疲劳。但 Agent 不判断业务合理性。它不知道"双重失效虽然功能正确但架构上应该收敛"，因为它不理解架构意图。

人工审查的核心价值是回答：**这段代码做了它该做的事，但有没有以正确的方式做？**

举一个本次没发生但可能发生的例子：Agent 审查通过了所有语法和模式检查，但如果 PostsService 里事件发布的时机放在了事务提交之前，而缓存监听器在事务回滚时已经生效了——这是一个**时序问题**，Agent 审查不会发现，因为它不会模拟"事务回滚同时缓存已清"这种跨组件交互。人工审查在读代码时会问："这个事件是在事务提交前发的还是之后？如果事务回滚了缓存已经被清了怎么办？"

### 3.2 人工功能复测

Agent 测试和人工测试的区别：

| | Agent 测试 | 人工测试 |
|------|-----------|---------|
| 方式 | 按预设脚本执行，逐条验证预期 | 探索性操作，直觉驱动 |
| 覆盖 | 预设场景内 100% 覆盖 | 场景外的意外发现 |
| 能发现 | 回归问题、修复是否生效 | 体验问题、边缘数据异常、交互问题 |
| 局限性 | 不会"试一下不按流程走会怎样" | 容易遗漏重复性验证 |

Agent 测试的价值是**保证预设场景不退化**。它的局限也在于此——Agent 不会问自己"换个顺序操作会不会有问题""参数填非法值会怎样""快速连续点击两次会怎样"。这些探索性行为是人天生就会的。

人工复测的核心价值是回答：**代码通过了所有预设测试，但作为一个用户，用起来正常吗？**

本次如果做人工复测，应该做的事：
- 打开浏览器，进入首页，确认侧边栏分类/标签/专栏正常显示
- 进入文章详情页，浏览一篇文章，返回首页确认热门文章排名是否变化
- 后台创建一篇文章，立即切回首页看侧边栏数据是否刷新
- 故意在后台连续快速创建和删除分类，看缓存是否出问题

这些不是脚本能预设的，是人在操作过程中的直觉反应。

### 3.3 验收的位置

验收位于整个流程的最后——不是在 Agent 审查和测试之间插入人工环节，而是**等 Agent 协作闭环全部完成后，人作为一个全新的角色介入**。

```
Agent 协作闭环（设计→开发→审查→测试→修复→复测）→ 人工验收（审查+复测）
```

这样的好处是：
- Agent 闭环已经保证了**正确性**（编译通过、测试全绿、审查问题已修复），验收时不纠缠低级问题
- 人验收时关注的是**合理性**和**体验**，而这些在 Agent 闭环里没有被覆盖
- 如果验收发现需要改，从修复阶段重新进入闭环

验收的标准不是"有没有 bug"，而是"能不能交付"。如果验收不通过，带着具体的发现回到修复阶段，再走一轮。

## 四、审查和测试的双层模型

总结前面的论述，审查和测试各自是**双层模型**：

```
Agent 审查（模式检查）→ 人工审查（语义审视）
Agent 测试（回归验证）→ 人工测试（探索验证）
```

第一层保证**对**——Agent 做全覆盖的机械化检查，不漏不掉。
第二层保证**好**——人做判断性的审视和探索性的操作，找机制覆盖不到的问题。

两层不可互相替代。只有 Agent 审查没有人工审查，放出去的是"语法正确但可能设计有问题"的代码。只有 Agent 测试没有人工测试，放出去的是"预设场景全过但可能在意外情况下崩溃"的功能。

---

## 五、设计文档是 Agent 协作的核心

本次产出了两类设计文档，它们在不同的协作阶段起到不同作用：

| 文档 | 用途 | 使用者 |
|------|------|--------|
| 架构设计 | 明确模块结构和技术选型 | dev-agent |
| 失效矩阵 | 穷举所有写操作 → 缓存映射 | dev-agent + tester-agent |

**设计文档 = Agent 的合约。** 人通过文档把需求翻译成精确的指令，Agent 按指令执行。合约越精确，执行越可靠。

## 六、Agent 能力边界

### 能做好

- **模式化编码**：4 个 CacheService 结构一致，Agent 不会遗漏也不会写歪
- **批量一致的修改**：7 个 Controller 的改法相同，Agent 全部覆盖
- **独立审查**：不被设计意图影响，从纯代码视角发现盲区
- **执行性测试**：按预设场景调 API、查日志、输出报告

### 做不好（必须人做）

- **跨模块设计决策**：两种方案各有权衡，Agent 不知道选哪个
- **优先级判断**：8 个问题修 3 个留 5 个，Agent 做不了
- **自主设计测试场景**：Agent 不会"想一下还有什么情况没测"
- **环境问题诊断**：class 文件过期这类问题，Agent 的排查能力有限
- **体验判断**：Agent 不会"试一下"，感受不到用起来顺不顺手、快不快
- **架构审视**：Agent 能检查语法，不能判断设计是不是最优解

---

## 七、方法论总结

1. **设计先行。** 不用急着写代码。一份精确的设计文档是 Agent 高质量的保证
2. **人和 Agent 各做各的。** 人做决策和验证，Agent 做执行和检查。别让 Agent 做决定，也别自己写重复代码
3. **Agent 协作闭环在前，人工验收在后。** Agent 保证正确性，人保证合理性和体验。顺序不能换
4. **审查和测试都是双层模型。** Agent 层做全覆盖检查，人层做判断和探索。两层不可替代
5. **修复需要人判断，复测需要新场景。** 审查报告是修复的输入，修复是复测的输入。闭环
