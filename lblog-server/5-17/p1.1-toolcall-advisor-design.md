## 废弃原因

1. **复杂度过高** — 过滤体系需要 advisor、registry、session manager 三个组件协同，而初期只需要"LLM 能加载领域指令"这个核心能力
2. **收益不明确** — 工具可见性控制在早期场景（只有 displayDiagramTool 和 loadSkillTool 两个工具）没有实际价值
3. **LLM 自主决策更灵活** — 让 LLM 决定何时加载技能，比后端硬编码过滤规则更适应变化
4. **阻碍迭代** — 每新增一个需隐藏的工具就要加注解 + 注册，开发体验差

## 当前方案的核心差异

| 维度 | 废弃方案 | 当前方案 |
|------|---------|---------|
| 技能控制 | 后端通过 advisor 拦截/过滤 tools | LLM 自主调用 loadSkill |
| 工具可见性 | @SkillTool 注解声明归属，未加载不可见 | 全部可见，LLM 自行选择 |
| 复杂度 | 高（3+ 组件） | 低（1 个 @Tool + CRUD） |
| 扩展性 | 加工具需要改注解 | 加工具直接注册即可 |
| 会话状态 | SkillSessionManager 维护 | 不维护（靠对话历史承载） |
