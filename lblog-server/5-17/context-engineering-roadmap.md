# AI 上下文工程落地路线图

> 日期：2026-05-17
> 适用项目：lblog-server AI 模块（DeepSeek + Spring AI 1.1.5）
> 目标：从零构建完整的上下文工程体系，逐步提升 AI 对话质量
> 设计原则：**循序渐进**——每个阶段独立可上线，后一阶段自然延伸前一阶段

---

## 学习路径总览

```ascii
第一阶段：让 AI 有记忆
  P0 对话持久化 → P1 上下文注入策略

  P6 评估与反馈 → P7 多代理架构

```

## 总体架构

```
┌──────────────────────────────────────────────────────────────┐
│                     用户请求 / API 层                         │
└─────────────────────────┬────────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────────┐
│                       ChatHistoryAdvisor  (✅ 已交付)          │
│  before(): 加载历史后拼入 prompt，不做压缩                    │
│  after(): 保存响应流到数据库                                  │
│                                                              │
│  ┌──────────────────────────────────────────────┐            │
│  │ CustomModelMessageConverter (模型消息转换)    │            │
│  │ DeepSeekMessageConverter (reasoning_content) │            │
│  └──────────────────────────────────────────────┘            │
└─────────────────────────┬────────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────────┐
│                    DeepSeekToolCallAdvisor  (✅ 已交付)        │
│  工具调用循环，每次递归调 chainCopy 中下一个 advisor           │
└─────────────────────────┬────────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────────┐
│                    CompressionAdvisor  (✅ P2 新增)            │
│  每次 LLM 调用前（首次 + 工具循环）检查上下文大小              │
│  ┌─────────────────────┐                                     │
│  │ CompressionStrategy  │ ← 策略模式，可替换                  │
│  │ SlidingWindowStrategy│ 按条数丢最旧消息，保留 system       │
│  └─────────────────────┘                                     │
└─────────────────────────┬────────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────────┐
│                    DeepSeek API                                │
└─────────────────────────────────────────────────────────────┘
```

**Advisor 链执行顺序：**
```
ChatHistoryAdvisor (HIGHEST_PRECEDENCE)
  → DeepSeekToolCallAdvisor (+2)
  → CompressionAdvisor (+3)   ← 每次 LLM 调用前执行
  → Model
```

**职责分离：**
- `ChatHistoryAdvisor` — 只做历史加载、消息保存、prompt 拼接，**不压缩**
- `DeepSeekToolCallAdvisor` — 只做工具调用循环
- `CompressionAdvisor` — 只做压缩，委托 `CompressionStrategy`
- `CompressionStrategy` — 策略接口，不同实现决定不同的压缩规则

## 第一阶段：让 AI 有记忆

### 🔴 P0 — 对话持久化

> 核心价值：从"无状态"到"有状态"，AI 对话可恢复、可回顾

[📄 p0-foundation.md](./p0-foundation.md)

**我们要解决什么：**
- 现在每次刷新页面，AI 就不记得之前聊了什么
- 消息都在前端内存里，后端什么都没存
- DeepSeek 的思考过程（reasoning_content）无处存放

**我们要做什么：**

| 任务 | 产出 |
|------|------|
| 建表 | `ai_chat_sessions` + `ai_chat_messages`（含 reasoning_content、tool_calls JSON） |
| Service + Mapper | 会话 CRUD、消息存储/加载 |
| `ModelMessageConverter` 接口 | 模型消息双向转换的抽象，加新模型只需加实现 |
| `DeepSeekMessageConverter` | DeepSeek 特有的 reasoningContent 处理 |
| `ChatHistoryAdvisor` | 自定义 advisor，自动加载历史/保存响应 |
| 改造 `DiagramService` | 从 advisor 拿历史，不再从请求体全量传递 |
| 改造前端 | 会话列表、历史加载、继续对话 |
| REST API | 会话列表/创建/删除/加载消息 |

**你将会学到：**
- Spring AI 的 Advisor 机制（before/after 拦截）
- 策略模式在模型适配中的应用
- SSE 流式响应的消息聚合与持久化
- 前端与后端的状态同步设计

**可以独立上线的功能：**
- ✅ 刷新页面后对话还在
- ✅ 可以查看/切换历史会话
- ✅ 前端展示 AI 的思考过程

### 🔴 P1 — 上下文注入策略

> 核心价值：控制"什么东西以什么形式进入上下文"，避免 token 浪费

**前置依赖：** P0（需要存储层来读写历史）

**我们要解决什么：**
- P0 把所有历史消息一股脑塞进上下文，token 浪费
- 不同场景需要不同的上下文（画图 vs 写代码）
- AI 的思考过程（reasoning_content）要不要进上下文
- 工具调用的中间结果太多，挤占窗口

**我们要做什么：**

| 任务 | 说明 | 涉及技术 | 状态 |
|------|------|---------|------|
| **分隔符设计** | 用 XML/Markdown 分隔系统指令、历史、文档、工具结果 | 注意力机制利用 | ⏳ |
| **技能式渐进加载** | 定义"上下文技能包"，按用户意图动态加载 | Level 0-4 分层注入 | ✅ 精简版已交付（skill=prompt） |
| **引用锚点** | 文档片段前加 `[1]` `[2]`，引导模型引用 | 减少幻觉 | ⏳ |
| **工具痕迹清理** | 只保留最终结果，丢弃中间失败的工具调用 | Token 节省 | ⏳ |
| **选择性注入** | reasoning_content 默认不进上下文 | Token 节省 | ⏳ |
| **结构化输出引导** | 需要 JSON 输出时 append Schema + few-shot | 输出质量 | ⏳ |
| **思考预算管理** | 简单问题限制思考深度，复杂问题充分思考 | DeepSeek 特定 | ⏳ |

**设计决策说明：** P1 当前只交付了"技能式渐进加载"一条，且做了方向调整——**去掉工具过滤体系**，采用 **Skill = Prompt** 模式（skill 只是一段 prompt 文本，LLM 通过 `loadSkill` tool 按需加载，不侵入 advisor 链）。其余 6 项均未启动，计划在后续迭代中逐步补齐。

---

#### 已交付：技能式渐进加载（精简版）

见独立设计文档：[p1-skill-design.md](./p1-skill-design.md)

**实际落地：**

| 任务 | 说明 | 实现 |
|------|------|------|
| **SkillPackage 持久化** | 技能包表结构 + MyBatis CRUD | SkillPackage domain + mapper + service |
| **loadSkill tool** | LLM 可调用的 `@Tool`，返回技能 prompt 文本 | LoadSkillTool.java |
| **技能列表提示** | System Prompt 中注入可用技能清单 | SkillSystemPromptBuilder.java |
| **按 agentType 筛选** | 不同 Agent 暴露不同技能包 | mapper.selectActiveByAgent() |
| **缓存** | 30s TTL Caffeine 缓存减少查库 | SkillServiceImpl Caffeine cache |

**关键设计：**

1. **Skill = Prompt** — 技能包没有 `tools`、`policy`、`modules` 等复杂字段，只有一段 `prompt` 文本。LLM 拿到 prompt 后自行遵循指令。
2. **不侵入 advisor 链** — 无 `SkillAwareToolCallAdvisor`，不拦截/过滤 tools。Skill 在 LLM 层通过 tool call 交互，与 advisor 链解耦。
3. **agentType 隔离** — `DrawChatRequest` 传 `agentType=draw`，`selectActiveByAgent("draw")` 只返回绘图相关技能。
4. **无意图识别** — 不实现语义分类器，LLM 自主决定何时/是否加载技能。

**可以独立上线的功能：**
- ✅ LLM 可动态加载领域指令，无需修改代码
- ✅ 不同 Agent（画图/聊天）看到不同的技能列表
- ✅ 技能包可热更新（30s 缓存过期）

## 第二阶段：让 AI 不遗忘

### 🟡 P2 — 上下文窗口管理

> 核心价值：在有限的上下文窗口中装下最多有效信息

**前置依赖：** P0（需要历史数据来管理）

**设计原则：** 压缩职责单一化——不分散在多个 advisor 中，统一由 `CompressionAdvisor` 委托 `CompressionStrategy` 处理。扩展新策略只需新增实现，不改 advisor。

---

#### 已交付

**架构：**

```
CompressionAdvisor (advisor，每次 LLM 调用前执行)
  ├ token 超预算 → 直接压缩（不问策略）
  └ token 没超   → 问 CompressionStrategy.shouldCompress()
       └ true → CompressionStrategy.compress()
            └ 例：SlidingWindowStrategy → 保留最近 N 条，丢最旧的
```

| 组件 | 职责 | 实现 |
|------|------|------|
| **`CompressionAdvisor`** | 上下文压缩入口，委托策略执行 | `ai/memory/advisor/CompressionAdvisor.java` |
| **`CompressionStrategy`** | 策略接口：`shouldCompress` + `compress` | `ai/memory/compression/CompressionStrategy.java` |
| **`SlidingWindowStrategy`** | 滑动窗口：消息数超阈值时保留最近 N 条 | `ai/memory/compression/SlidingWindowStrategy.java` |
| **`TokenEstimator`** | token 估算接口 | `ai/memory/estimator/TokenEstimator.java` |
| **`CharBasedTokenEstimator`** | 字符估算实现（`len/2`）,可替换为 jtokkit | `ai/memory/estimator/CharBasedTokenEstimator.java` |

**关键设计（策略模式）：**

```java
// CompressionAdvisor 只依赖接口，不碰具体策略
private ChatClientRequest compressIfNeeded(ChatClientRequest request) {
    if (overBudget || strategy.shouldCompress(messages)) {
        List<Message> result = strategy.compress(messages);
        // ...
    }
}
```

**配置方式（AiConfig）：**

```java
@Bean
public CompressionStrategy compressionStrategy() {
    return new SlidingWindowStrategy(50, 70);
    // 参数：压缩到 50 条，超过 70 条触发
}
```

换策略只需改这一处，不用动 advisor。

#### 待实现（后续可选）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| **Token 感知压缩** | 按 token 而非条数裁切（需 jtokkit） | 低 |
| **新鲜度评分** | 按消息新鲜度加权，旧的优先丢 | 低 |
| **摘要压缩** | 旧对话 → LLM 压缩为摘要替代原文 | 低 |

### 🟡 P3 — 检索增强 RAG

> 核心价值：当 AI 不知道答案时，能主动查资料

**前置依赖：** P0（基础存储），可与 P1、P2 并行

**我们要解决什么：**
- AI 不能凭空知道你的项目代码、数据库结构、业务文档
- 全量文档不能塞进上下文（token 不够）
- 需要一种机制：用户提问 → 找到最相关的知识 → 注入上下文

**我们要做什么：**

| 任务 | 说明 | 涉及技术 |
|------|------|---------|
| **知识库设计** | 确定哪些内容入库（项目文档、API 文档、配置说明） | 知识管理 |
| **语义分块** | 按段落/章节边界分块，不截断语义 | 分块策略 |
| **向量存储** | 引入 embedding 模型 + MySQL vector 存储 | 向量数据库 |
| **小到大检索** | 索引小块保证召回，提供所属大块保留语境 | 检索精度 |
| **混合搜索** | BM25 精确匹配 + 向量语义搜索，加权融合 | 混合检索 |
| **查询改写** | 用户模糊问题 → LLM 改写为精确检索关键词 | 查询优化 |
| **重排序** | 初检结果用交叉编码器重排，最相关置顶 | 排序优化 |
| **元数据过滤** | 按时间、来源、标签精确过滤 | 精准检索 |

**RAG 完整流程：**

```
用户提问: "数据库的用户表有哪些字段"
        │
        ▼ 查询改写
  "用户表字段 schema" → "users table columns"
        │
        ▼ 混合检索
  ├─ BM25: "users table"、"column" → 精确匹配
  └─ 向量: [embedding] → 语义相似度
        │
        ▼ 融合排序 → 取 top 5 → 重排序
        │
        ▼ 元数据过滤 → 排除过时版本
        │
        ▼ 注入上下文
  [1] CREATE TABLE users (id BIGINT, name VARCHAR...)
  [2] users 表包含 10 个字段，主键为 id...
        │
        ▼ 模型回答
  "用户表有这些字段：id（主键）、name（用户名）、email..."
```

**你将会学到：**
- Embedding 模型的基本原理
- 向量检索 vs 关键词搜索的取舍
- RAG 的核心流程（检索 → 排序 → 注入）
- 查询改写的 Prompt 设计

**可以独立上线的功能：**
- ✅ AI 能查询项目文档回答具体问题
- ✅ 检索结果附引用来源
- ✅ 模糊问题能被理解并找到正确答案

## 第三阶段：让 AI 更懂你

### 🟢 P4 — 用户长期记忆

> 核心价值：跨会话记住用户偏好和关键事实，越用越懂用户

**前置依赖：** P0（存储） + P3（检索能力）

**我们要解决什么：**
- 每次新会话 AI 都不认识用户
- 用户说了自己的偏好（"我喜欢简洁风格"），下次就忘了
- 历史对话中的关键信息没有被提取和复用
- 旧信息和新信息冲突时不知道信哪个

**我们要做什么：**

| 任务 | 说明 | 涉及技术 |
|------|------|---------|
| **记忆提取器** | 用 LLM 从对话中自动提取关键信息 → key-value 存入记忆表 | 结构化记忆 |
| **长期记忆库** | 基于 P3 的向量存储能力，存储和检索记忆 | 向量检索 |
| **记忆注入路由** | 判断当前问题是否需要加载长期记忆，避免每次都塞无用信息 | 路由决策 |
| **冲突消解** | system prompt 加规则"以用户最新陈述为准" | 一致性保障 |
| **时间感知** | 注入当前日期、用户时区、会话持续时间 | 时效性 |
| **上下文隔离** | 切换话题时清空不相关历史，新话题从零开始 | 话题管理 |
| **动态提示词** | 不同用户角色、不同任务挂载不同指令模块 | 个性化 |

**记忆提取流程：**

```
对话进行中...
        │
        ▼ MemoryExtractor（异步，对话空闲时触发）
  分析最近几轮对话
        │
  ├─ 提取事实性信息：
  │   "我叫张三"           → {key: user_name, value: 张三}
  │   "我喜欢极简风格"     → {key: diagram_style, value: minimal}
  │   "我用 Java 17"       → {key: tech_stack, value: java17}
  │
  └─ 存入 ai_user_memories 表
        │
        ▼ 下次对话开始时
  → 检索该用户的所有活跃记忆
  → 注入 system prompt:
    "关于你已了解的信息：用户偏好极简风格图表，使用 Java 17..."
```

**上下文隔离（话题切换）：**

```
用户: "帮我画一个 ER 图"   → 话题A: 绘图
  上下文: 绘图规范 + 图表 XML + 绘图历史

用户: "今天天气怎么样"    → 话题B: 天气（与绘图无关）
  上下文: 重置，不包含绘图历史
  但长期记忆仍保留"用户画过 ER 图"这个事实

用户: "继续画刚才的图"    → 回到话题A
  上下文: 恢复绘图历史
```

**可以独立上线的功能：**
- ✅ 第一次告诉 AI 你的名字，第二次它还记得
- ✅ 偏好设置跨会话持久化
- ✅ 不同话题互不干扰

### 🟢 P5 — 缓存与复用

> 核心价值：减少 LLM 调用次数，降低延迟和成本

**前置依赖：** P0（基础存储），可在任何阶段独立开始

**我们要做什么：**

| 任务 | 说明 |
|------|------|
| **语义缓存** | 相似问题（如 FAQ）命中缓存直接返回，不走 LLM |
| **预编译上下文片段** | 常用工具定义、函数签名预编译成 token 高效的片段 |
| **Prompt Caching** | 将不变的系统 prompt 放前面，利用 API 的缓存能力 |

**可以独立上线的功能：**
- ✅ 相同或相似问题秒回
- ✅ API 调用次数减少 30%+

## 第四阶段：让 AI 可度量

### 🟢 P6 — 评估与反馈

> 核心价值：用量化指标持续改进上下文质量

**前置依赖：** P0-P5（需要有完整系统才能评估）

**我们要做什么：**

| 任务 | 说明 |
|------|------|
| **相关性评估** | 每次请求后评估注入的文档/记忆是否相关，分数低的优化 |
| **忠实度检查** | 检查回答是否基于上下文而非幻觉 |
| **Token 效率指标** | 统计每 K token 产出的有效信息量 |
| **A/B 测试** | 同任务不同 prompt/策略对比效果 |
| **用户反馈闭环** | 点赞/点踩 → 反向优化检索策略和压缩强度 |

### ⚪ P7 — 多代理架构

> 核心价值：通过代理拆分解决单一上下文过载问题

**前置依赖：** P0-P4，可选

**我们要做什么：**

| 任务 | 说明 |
|------|------|
| **规划-执行分离** | 主代理做规划，子代理携带精简上下文执行具体任务 |
| **上下文窗口隔离** | 长期规划与短期执行使用不同的上下文窗口 |
| **辩论审查** | 多个代理用不同视角审阅同一材料，再合并结论 |

## 实施建议

1. **按顺序推进**：P0 → P1 → P2 → P3 → P4，每个阶段都建立在前一阶段之上
2. **每阶段可独立上线**：不用全做完才发布，每个阶段都有独立价值
3. **P3 可提前**：如果检索需求迫切，P3 可以提前到 P1 完成后并行做
4. **P5/P7 随心**：缓存和多代理是锦上添花，按需开始
5. **P6 是长跑**：评估机制可以逐步建立，不要等到最后才做

### 推荐的学习节奏

```
第一个月: P0 + P1     → 有记忆的 AI
第二个月: P2 + P3     → 不遗忘、能检索
第三个月: P4          → 个性化
持续优化:  P5 + P6 + P7
```
