# 03 — 对话持久化（P0）

## 解决了什么问题

最开始 AI 绘图每次请求是无状态的——前端把所有历史消息全量塞进请求体，后端不存任何东西。刷新页面对话消失，没法查看历史会话。

P0 做三件事：
1. **存下来** — 每次对话完整存入 MySQL（会话 + 消息）
2. **读回去** — 下次请求自动从 DB 加载历史注入上下文
3. **选着用** — reasoning_content（思考过程）按策略决定是否注入，减少 token 浪费

---

## 数据模型

```
ai_chat_sessions                    ai_chat_messages
┌──────────────────────┐           ┌──────────────────────────┐
│ id                   │←──────────│ session_id (FK, CASCADE)  │
│ user_id              │           │ role (user/assistant/tool)│
│ agent_type (draw/chat)│          │ content (LONGTEXT)        │
│ title                │           │ reasoning_content         │
│ model_name           │           │ tool_calls (JSON)         │
│ status (1/-1)        │           │ tool_call_id              │
│ created_at           │           │ name                      │
│ updated_at           │           │ msg_index (严格递增序号)    │
└──────────────────────┘           │ tokens                    │
                                   │ metadata (JSON预留)        │
                                   └──────────────────────────┘
```

设计要点：

- **msg_index 而非 timestamp 排序** — 消息顺序不依赖时钟，并发安全靠序号。每条新消息 = 当前 max_index + 1
- **tool_calls JSON 列** — 标准 OpenAI function calling 格式存储，各模型通用
- **reasoning_content 统一** — DeepSeek 叫 reasoning_content，Claude 叫 thinking，存储层统一为一个字段
- **软删除** — status = 1 活跃，0 归档，-1 删除。物理不删，可恢复
- **ON DELETE CASCADE** — 删会话自动级联删消息
- **messageCount 不冗余存储** — 会话列表查询时通过 `LEFT JOIN + COUNT(messages)` 实时计算，避免计数维护不一致

## 核心流程

### 写入路径（每次 LLM 调用）

```
DiagramService.chatStream()
  → ChatClient.stream()
    → Advisor 链
      → ChatHistoryAdvisor.before()
          │
          ├─ loadingStrategy.load(sid, store)  ← 1. 按策略加载（无策略则全量降级）
          │     └─ SlidingWindow: 只取最后 20 条
          │
          ├─ saveCurrentUserMessage()          ← 2. 先存当前用户消息
          │     └─ msg_index = maxIndex + 1
          │
          ├─ buildContextMessages()            ← 3. DB entity → Spring AI Message
          │     ├─ assistant → DeepSeekMessageConverter.toContextMessage()
          │     ├─ user      → new UserMessage()
          │     └─ tool      → 跳过（DeepSeekToolCallAdvisor 会重建）
          │
          └─ 组装 prompt                        ← 4. system → 历史 → 当前 → LLM
              
      → DeepSeekToolCallAdvisor (工具调用，可能新增消息)
      
      → CompressionAdvisor (兜底：工具调用结果膨胀 → 继续裁)
      
      → ChatHistoryAdvisor.after()
          │  (非流式：直接保存)
          │  (流式：doOnComplete 时拼装 chunks → 保存)
          │
          ├─ converter.toStorageMessage() / toStorageMessageFromChunks()
          └─ chatMemoryStore.saveMessages()
```

### 读取路径（恢复历史会话）

```
ChatSessionController.loadMessages(id)
  → messageService.loadMessages(id)
    → SELECT * FROM ai_chat_messages WHERE session_id=? ORDER BY msg_index ASC
  → convertToVOList()
  → 返回前端渲染
```

消息加载到上下文时，via ChatHistoryAdvisor.before() 走同样的 `loadHistory → buildContextMessages` 路径。

## 关键设计评审

每个设计决策按 **当前做法 → 为什么 → 替代方案 → 推荐** 结构展开。

---

### 决策 1：为什么不用 Spring AI 自带的 JdbcChatMemoryRepository

Spring AI 内置了基于 JDBC 的对话记忆存储。但没有用它。

| 对比维度 | Spring AI 自带 | 自建方案 |
|---------|---------------|---------|
| 表结构 | 4 列：conversation_id, content, type, timestamp | 扩展：reasoning_content, tool_calls JSON, msg_index, metadata |
| 模型适配 | 只存 content 文本，丢弃各模型特殊字段 | ModelMessageConverter 策略模式，DeepSeek 的 reasoning 不会丢 |
| 消息顺序 | 依赖 timestamp（并发下可能重排） | msg_index 显式序号 |
| 扩展性 | 表结构固定 | metadata JSON + converter 接口，加模型只需再加实现 |

**评价：** 不用是对的。Spring AI 的 JdbcChatMemoryRepository 是为"演示"设计的，不是为"生产"设计的。对于需要存 DeepSeek reasoning_content、工具调用 JSON、支持多模型切换的场景，自建方案的表结构和 converter 策略模式是必要的。

---

### 决策 2：sessionId 的传递方式

**当前做法：** 通过 Spring AI 的 `ChatClientRequest.context()` (即 advisor params) 传递：

```java
// DiagramService 中
ChatClient prompt = chatClient.prompt()
    .advisorParams(spec -> spec
        .param("sessionId", sessionId)
        .param("modelName", modelName));

// ChatHistoryAdvisor 中获取
Map<String, Object> ctx = request.context();
String sessionId = (String) ctx.get("sessionId");
```

**为什么这样：** Spring AI 的 context 在 advisor 链中自动透传，不需要 ThreadLocal。`ChatClientRequest.context()` 在整个调用链中保持一致。

**替代方案：**

| 方案 | 做法 | 问题 |
|------|------|------|
| A. Advisor context（当前） | 通过 Spring AI 的 context Map 传递 | 需要手动塞入，类型不安全（到处都是 String cast） |
| B. ThreadLocal | 存到当前线程 | Reactor 流式场景跨线程失效（已发生过教训） |
| C. 自定义 Request 包装 | 把 sessionId 编码到 Prompt options 中 | Spring AI 的 options 不为此设计 |
| D. Advisor 构造注入 | 把 sessionId 作为 advisor 实例变量 | advisor 是单例，多请求会互相覆盖 |

**评价：** A 是 Spring AI 框架内最干净的方案，也是官方推荐的做法。缺点只是类型不安全（String cast），可以接受。ThreadLocal 方案在这个项目早期用过，流式场景翻车后切到现在的方案——教训正确吸取了。

---

### 决策 3：ModelMessageConverter 策略模式

**当前做法：** 接口定义 `toContextMessage()`（DB→Spring AI）和 `toStorageMessage()` / `toStorageMessageFromChunks()`（Spring AI→DB）。每种模型一个实现。ChatHistoryAdvisor 根据 modelName 自动路由：

```java
private ModelMessageConverter findConverter(String modelName) {
    return converters.stream()
        .filter(c -> c.supports(modelName))
        .findFirst()
        .orElse(converters.get(0)); // 兜底：第一个注册的 converter
}
```

**为什么这样做：** 不同 LLM 的 Message 对象结构不同——DeepSeek 有 `reasoningContent` 字段，Claude 的 thinking 在另外的地方。converter 把这些差异隔离在各自实现里，ChatHistoryAdvisor 不需要知道具体模型。

**替代方案：**

| 方案 | 做法 | 加模型成本 |
|------|------|----------|
| A. 策略模式（当前） | 接口 + 模型实现 | 新建一个类 |
| B. if-else 在 advisor 里 | instanceof 判断 Message 类型 | 改 advisor 代码 |
| C. 统一存储格式 | 所有模型的消息都在存储前归一化为通用格式 | 信息损失（模型特定字段如 reasoning 被迫丢弃） |

**评价：** 策略模式在这里用得正确。未来加 Claude/OpenAI 只需新建一个实现类，零改动现有代码。

一个可改进的点：`supportedModels()` 方法返回具体模型名列表（如 `deepseek-chat`, `deepseek-v4-flash`），但 `supports()` 用的是 `startsWith("deepseek")` 前缀匹配。这两者重复了——要么只留 `supports()` 删除 `supportedModels()`，要么反过来让 `supports()` 委托给 `supportedModels().contains(modelName)`。

### 决策 4：reasoning_content 的选择性注入

**当前做法：** ChatHistoryAdvisor 从 `site_config` 表读取 `reasoning_inject` 配置键，决定历史中的 reasoning_content 是否注入上下文：

```java
private ContextPolicy buildPolicy() {
    if ("true".equals(siteConfigMapper.selectConfigValue("reasoning_inject"))) {
        policy.setIncludeReasoning(true);
    }
}
```

**为什么这样做：** reasoning_content 是 AI 的"思考草稿"——对用户有价值（可折叠展示），但对 LLM 本身没意义（下次调用不需要知道上次怎么想的）。默认不注入节省 token。但当需要"连贯思考"的长任务场景时，可以通过配置打开。

**替代方案：**

| 方案 | 做法 | Token消耗 | 适用场景 |
|------|------|----------|---------|
| A. 默认不注入 + 配置开关（当前） | site_config 控制 | 低 | 大部分对话 |
| B. 始终注入 | 把 reasoning_content 拼到 assistant 消息前 | 高（白白浪费 token） | 几乎不需要 |
| C. 摘要后注入 | 用 LLM 把 reasoning 压缩成一句话再注入 | 中（需额外 API 调用） | 需要 AI 了解"之前的思考方向" |
| D. 智能注入 | 根据当前问题类型判断：简单问答不注入，复杂推理注入 | 中 | 理想态但实现复杂 |

**评价：** A 是务实选择。当前只有 draw 一个 Agent 时够用。未来如果有多 Agent，建议 `reasoning_inject` 从全局 key 改为按 agent_type 的 key（如 `reasoning_inject.draw` / `reasoning_inject.chat`），不同 Agent 的需求可能不同。

### 决策 5：Conversation 与 Memory 的分层

**当前做法：**

```
ai/conversation/           ← 存储层（纯 CRUD，不感知 Spring AI）
  ├── domain/              ChatSession + ChatMessage（MyBatis 实体）
  ├── mapper/              ChatSessionMapper + ChatMessageMapper
  ├── service/             ChatSessionService + ChatMessageService
  └── controller/          REST API /api/v1/ai/chat/sessions

ai/memory/                 ← 记忆层（理解 Spring AI，协调 converter + advisor）
  ├── ChatMemoryStore      接口：loadHistory / saveMessages / getOrCreateSession
  ├── ChatMemoryStoreImpl  实现：委托给 conversation 的 Service
  ├── advisor/             ChatHistoryAdvisor + CompressionAdvisor
  ├── converter/           ModelMessageConverter 接口 + DeepSeek 实现
  └── compression/         压缩策略
```

**设计意图：** conversation 层不知道 advisor 的存在，memory 层不直接碰 MyBatis Mapper。两层通过 ChatMemoryStore 接口隔离。

**实际情况：** ChatMemoryStoreImpl 是一个薄到几乎透明的委托层：

```java
@Override
public List<ChatMessage> loadHistory(Long sessionId) {
    return messageService.loadMessages(sessionId);
}
```

没有额外的转换、聚合、缓存逻辑。这个接口的价值（解耦）在当前规模下尚未体现。

**评价：** 分层方向是对的，但当前只有一个实现（MySQL），没有需要隔离的变化点。不是过度设计——是"现在用不上，但放对了地方"。未来如果要把历史存储换成 Redis，只改 ChatMemoryStoreImpl，conversation 和 memory 两层都不受影响。

### 决策 6：LoadingStrategy — 可插拔的加载策略

**当前做法：** `LoadingStrategy` 接口只有一个方法：

```java
List<ChatMessage> load(Long sessionId, ChatMemoryStore store);
```

ChatHistoryAdvisor 注入它后按策略加载历史，未注入则降级全量。与压缩无关——它只管"从 DB 取什么"。

**为什么这样做：** 之前的链路"全量加载 50 条 → 转换 → CompressionAdvisor 丢 30 条"是浪费。把截断逻辑前置到加载阶段，DB 取出来的就是已经截好的。

**压缩策略为什么实现了它：** 滑动窗口知道自己只关心最后 20 条，所以 `load()` 只取最后 20 条——DB 不会多查。未来摘要压缩知道自己需要"最后 N 条完整 + 其余摘要"，load() 也一起搞定。**每个压缩策略天然决定了怎么加载，所以实现了同一个接口，但不代表接口本身依赖压缩。**

ChatHistoryAdvisor 不关心传入的是滑动窗口、摘要压缩还是什么都不传——它只调 `load()`。

**替代方案：**

| 方案 | 做法 | 问题 |
|------|------|------|
| A. 双面策略（当前） | LoadingStrategy 接口，CompressionStrategy 继承 | — |
| B. 两个独立策略 | LoadingPolicy + CompressionStrategy 分开注册 | 可能错配：摘要加载 + 窗口压缩，语义矛盾 |
| C. LoadPlan 中间层 | 策略返回 LoadPlan{fullCount, summaries}，advisor 解读 | 多余间接层，策略不直接干活 |

**评价：** 加载策略是一个独立概念，不和压缩绑定。ChatHistoryAdvisor 拿到什么策略就用什么加载，降级则全量。未来即使不用压缩，也可以单独注入一个 FixedWindowLoadingStrategy(N) 控制加载量。

## 实现层面发现的问题

### ✅ Bug 1：message_count 和 total_tokens 永远是 0（已修复）

**问题：** `ChatSessionService.updateStats()` 定义了（interface + impl + mapper + XML 全有），但 ChatHistoryAdvisor 从未调用它。`ChatMemoryStoreImpl.touchSession()` 是空方法体。`message_count` 和 `total_tokens` 初始化是 0，之后永远不变。

**修复方案：** 删除冗余字段，会话列表查询改为 LEFT JOIN + COUNT 实时计算。`total_tokens` 等后续 P6 评估阶段有实际需求时再按需加回。

**涉及改动（8 个文件）：**
- `ChatSession.java` — 删除 messageCount、totalTokens 字段
- `ChatSessionMapper.java` — 删除 updateStats；selectByUserAndAgent 直接返回 ChatSessionVO
- `ChatSessionMapper.xml` — 删除 updateStats SQL；INSERT 不再写两列；列表查询改为 LEFT JOIN + COUNT + GROUP BY
- `ChatSessionService.java` — 删除 updateStats
- `ChatSessionServiceImpl.java` — 删除 updateStats 实现；listSessions 直接返回 mapper 结果
- `ChatMemoryStore.java` — 删除 touchSession
- `ChatMemoryStoreImpl.java` — 删除 touchSession 空实现
- `ChatHistoryAdvisor.java` — 删除两处 touchSession() 调用

### 🔴 Bug：previewText 从未赋值

`ChatSessionVO` 定义了 `previewText` 字段（会话列表的最后一条消息预览），但 ChatSessionController 的 `convertToVO()` 从未设置它。前端拿到的 `previewText` 永远是 null。

**修复方案：** 在会话列表查询时，关联查最后一条 assistant 消息的前 100 个字符作为 previewText。或者改 SQL 用 JOIN + SUBSTRING。

### 🟡 三个 Service 方法从未被调用

| 方法 | 定义位置 | 调用方 |
|------|---------|--------|
| `saveUserMessage()` | ChatMessageService | ⚠️ 无调用方 |
| `saveAssistantMessage()` | ChatMessageService | ⚠️ 无调用方 |
| `saveToolMessage()` | ChatMessageService | ⚠️ 无调用方 |
| `loadRecentMessages()` | ChatMessageService | ⚠️ 无调用方 |

设计文档里设计了"单条保存"接口，但实际实现时 ChatHistoryAdvisor 全部走 `batchSave()`。这 4 个方法应该要么删掉，要么在需要的地方用起来（比如未来 admin 手动插入消息）。

### 🟡 getOrCreateSession 总是创建新会话

```java
// ChatMemoryStoreImpl.getOrCreateSession()
@Override
public ChatSession getOrCreateSession(Long userId, String agentType, String modelName) {
    return sessionService.createSession(userId, agentType, modelName); // 只管创建
}
```

方法名叫 "get or create"，但实际逻辑是 "always create"。如果用户刷新页面后重新发请求、又没带 sessionId，DiagramService 会再调 getOrCreateSession，然后**同一个用户对同一个 agent 每次刷新都创建新会话**。

**修复方案：** 先查是否存在该用户+该 agent 的活跃会话，有则返回，没有再创建。

---

### 🟡 会话标题是硬编码的

```java
session.setTitle("New " + agentType + " session");
// → "New draw session"
```

不包含任何上下文信息（如第一条用户消息的摘要）。好处是简单，坏处是会话列表里全是"New draw session"，用户区分不了。

**替代方案：**

| 方案 | 做法 | 成本 |
|------|------|------|
| A. 硬编码（当前） | "New draw session" | 零 |
| B. 用户第一条消息截断 | 创建会话时用第一条消息前 30 字做标题 | 低，改 createSession 传入首条消息 |
| C. LLM 自动生成标题 | 第一次回复完成后异步调 LLM 生成标题 | 中，一次额外 API 调用 |
| D. 用户手动编辑 | 已有 updateTitle API，纯前端行为 | 零（API 已有） |

**评价：** B + D 是最务实的组合——创建时用首条消息截断做默认标题，UI 允许用户双击编辑。
