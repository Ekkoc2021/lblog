# 01 — 架构总览与评审

## 一、整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                         transport                                │
│  DiagramController  ←── SSE (text-delta / reasoning / tool-call)  │
│  ChatSessionController ←── REST JSON                              │
│  AdminPromptController ←── REST JSON (管理端)                      │
└────────────────────────────┬─────────────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────────────┐
│                       orchestration                              │
│  DiagramService (@Async 虚拟线程)                                  │
│  → 组装 system prompt + 注册 tools                                │
│  → ChatClient.stream() / .call()                                 │
│  → 心跳管理 + SSE 事件发送                                        │
└────────────────────────────┬─────────────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────────────┐
│                     advisor chain (interceptor)                   │
│  ChatHistoryAdvisor → DeepSeekToolCallAdvisor → CompressionAdvisor │
│  加载/保存历史          工具调用 + reasoning 捕获     token 预算控制 │
└────────────────────────────┬─────────────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────────────┐
│                     infrastructure                               │
│  prompt/   —— 提示词 (DB→文件→硬编码)    conversation/ —— 会话/消息  │
│  skill/    —— 技能包 (loadSkill tool)   memory/      —— 压缩+转换  │
└──────────────────────────────────────────────────────────────────┘
```

分层清晰：**transport → orchestration → advisor chain → infrastructure**。每层对下层是调用方，对上层是透明提供方。

## 二、做得好的地方

### 2.1 Advisor 链的职责分离

三个 advisor 各管一件事，没有互相渗透：

- ChatHistoryAdvisor：只管存取，**加载时按 LoadingStrategy 截断**，不推给下游
- DeepSeekToolCallAdvisor：只管工具调用循环 + reasoning 捕获
- CompressionAdvisor：只管 token 预算兜底（工具调用结果膨胀等不可预知情况）

对比很多项目把"加载历史 + 压缩 + 工具调用"写在一个 500 行的 service 方法里，这个分离在可测试性和可维护性上是成熟的。

### 2.2 策略模式的三处正确使用

**LoadingStrategy（加载策略）：** 接口只有一个方法 `load(sessionId, store) → List<ChatMessage>`，ChatHistoryAdvisor 注入后按策略取历史，未注入则降级全量加载。SlidingWindowStrategy 实现它只取最后 20 条，未来 SummaryStrategy 可实现它返回"完整消息 + 摘要"。**这只是一个加载接口，和压缩无关。**

**CompressionStrategy（压缩策略）：** 继承 LoadingStrategy。因为每种压缩方式天然知道自己希望怎么加载——滑动窗口希望只加载最后 N 条，摘要压缩希望加载完整消息 + 摘要。同一个对象管"加载时取什么"和"运行时裁什么"，语义对齐不做错配。这是开闭原则的又一例——换压缩策略时加载行为自动跟着变。

**ModelMessageConverter**（模型适配）：接口隔离不同 LLM 消息格式差异，加新模型加一个实现类。

### 2.3 Skill = Prompt 的简化决策

从"注解 + Registry + Advisor + 会话管理"6 个组件的重方案，收敛到"一个 @Tool + 一段 prompt 文本"。这是一个好的架构判断——在需求不明确时选择最轻的方案，而不是为"可能"需要的工具过滤能力提前买单。

### 2.4 Conversation 与 Memory 的分层

`ai/conversation/` 只管存储（CRUD），`ai/memory/` 管怎么把存储的东西变成 LLM 上下文。ChatMemoryStore 接口隔离了两层，conversation 层不知道 advisor 的存在，memory 层不直接碰 MyBatis。这个分层虽然薄（ChatMemoryStoreImpl 只是委托），但方向是对的——未来换存储方案（如 Redis）只改 memory 层。

### 2.5 提示词 DB 优先 + 文件兜底

不引入配置中心、不依赖外部服务，用已有的 MySQL + classpath 文件就实现了"生产热修改 + 开发免查库"。务实。

## 三、架构层面的问题

### 3.1 DrawService 做了太多事

**已做：** 传输层（心跳 + SSE 格式化 + 断连检测）已剥离为 `AgentStreamTransport` 接口 + `SseStreamTransport`。DrawService 不再直接操作 SseEmitter 和 ScheduledExecutorService，只调 `transport.sendXxx()`。未来 WebSocket 只需加 `WebSocketStreamTransport`。

**待做：** 编排骨架（调 ChatClient、处理 chunk）仍只在 DrawService 里。等加第二个 Agent 时提取 `AgentExecutor` 基类。

### 3.2 Tool 与传输层耦合

`DisplayDrawTool` 通过 `ToolContext` 拿 `SseEmitter`，工具逻辑和 SSE 捆在一起。非 SSE 场景无法复用，单元测试必须 mock emitter。

当前只有一个 Agent、一个 tool，耦合未造成实际 bug。等加第二个 Agent 时，工具怎么和传输层交互这个问题会有更明确的抽象需求——到那时再解耦。

### 3.3 PromptManager 依然在做模型路由

**现状：** PromptManager 通过字符串匹配选择提示词版本：

```java
if (modelId.contains("claude-opus-4-5")) { ... }
```

提示词内容已经外置了（从 Java 常量移到 DB+文件），但**选哪个提示词的逻辑**还是写在代码里。每加一个新模型要改 Java 代码，这和外置化的初衷矛盾。

**建议：** 在 `ai_prompts` 表或 `site_config` 中建一个 `model → prompt_key` 的映射，PromptManager 只管"给我模型名，我查出 key，调 AiPromptService 拿内容"，不做任何 if-else 模型判断。

### 3.4 缺少 Agent 抽象

**现状：** 如果现在要加第二个 Agent（如 AI 聊天），需要：

1. 复制 DiagramService 的编排结构（chatStream 方法）
2. 新建 Controller、Request DTO、PromptManager
3. 复制一套 tool 注册 + toolContext 设置逻辑
4. 在 AiConfig 中再配一个 ChatClient Bean

**根本原因：** 没有抽象出"Agent 执行器"的概念。当前代码中 Agent 的概念散落在各个地方：

- `agent_type` 字段（数据库层已有，但只用于 conversation 过滤）
- `toolContext("agentType", "draw")`（运行时传入，但只在 LoadSkillTool 中用到）
- 各 Agent 独享的 PromptManager 和 Controller

**建议：** 提取一个 `AgentExecutor` 基类/接口，定义"一个 Agent 执行一次对话"的标准流程：

```
AgentExecutor.execute(request):
  1. 构建 system prompt（委托 AgentPromptBuilder）
  2. 注册 tools（委托 ToolRegistry）
  3. 调用 ChatClient（统一 advisor 链）
  4. 处理结果（委托 ResultHandler）
```

DiagramService 变成 DrawAgentExecutor 继承这个骨架，只覆写 step 1 和 step 2。

### 3.5 Token 估算精度低

**现状：** `CharBasedTokenEstimator` 用 `长度 / 2` 估算。对于中文文本（一个字符可能是一个 token），误差可能到 50-100%。在大模型 token 计数中，中英文的实际 token 差异很大。

**影响：** `CompressionAdvisor` 基于这个粗略估算做压缩决策——如果预估偏低，可能实际已经超了还没触发压缩；预估偏高，可能不该压缩的也压了。

**建议：** 引入 `jtokkit`（Java tokenizer，支持 cl100k_base 编码）替换字符估算。或者至少改为"长度 / 3.5"（对中文更保守的估算）。更好的方案是在 `CompressionAdvisor` 中加一个安全边际系数（如实际阈值 = 估算阈值 × 0.8）。

### 3.6 异常处理缺口

**现状：**
- DeepSeek API 不可用时，DiagramService 的 catch 块只打日志 + emitter.completeWithError，没有重试
- 没有熔断机制——如果 API 连续失败，请求还会继续发
- 没有降级路径——比如切换到另一个模型或返回缓存结果

**建议：** 至少加一层 Spring Retry（指数退避 2-3 次）。更完整的方案是引入 Resilience4j CircuitBreaker，在 AiConfig 中给 ChatClient 包装一个熔断装饰器。

### 3.7 Advisor 链的隐式依赖

**现状：** Advisor 链的顺序靠 `@Order` 数值控制：

```java
ChatHistoryAdvisor:     HIGHEST_PRECEDENCE
DeepSeekToolCallAdvisor: +2
CompressionAdvisor:     +3
```

如果未来有人加第四个 advisor 用了 `+1` 的 Order，它会被插到 ChatHistoryAdvisor 和 DeepSeekToolCallAdvisor 之间，可能导致微妙 bug（比如新增 advisor 修改了消息结构，工具调用 advisor 后续解析出错）。

**建议：** 在 AiConfig 中显式列出 advisor 顺序（而不是靠数字），并对每个 advisor 的输入/输出约定做注释说明。或者用一个封装类 `AdvisorChain` 强制按固定顺序追加。

## 四、总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 模块边界 | ⭐⭐⭐⭐ | conversation / memory / prompt / skill 各司其职 |
| 扩展性 | ⭐⭐⭐⭐ | 策略模式三处用对，LoadingStrategy 让换压缩策略语义对齐 |
| 简洁性 | ⭐⭐⭐⭐ | Skill = Prompt 是好的简化决策 |
| 健壮性 | ⭐⭐ | 缺重试、熔断、token 估算精度低 |
| 可测试性 | ⭐⭐ | Tool 耦合 SSE 传输层，单元测试难以隔离 |
| 一致性 | ⭐⭐⭐ | 少数硬编码残留（PromptManager 模型路由、DrawRateLimiter） |

> 一句话：架构骨架是好的，分层和策略模式用得克制而得当。主要债务在传输层耦合和 Agent 抽象缺失——这两个都是在加第二个 Agent 时会立刻撞上的墙。
