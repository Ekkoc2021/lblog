# 09 — 未来方向：上下文工程在 Spring AI 中的实践

前面八篇文档完成了 P0-P2：对话持久化、Skill 系统、窗口压缩、模型适配、传输层抽象——一套完整的上下文工程骨架。这篇从学习角度盘点还有哪些值得在这个项目里继续探索的主题，以及每个主题在 Spring AI 下怎么落地。

---

## 第一梯队：有明确 API 支持，改配置或少量代码即可

这些不是"设想"——API 已经支持，只是项目没启用。

### 工具搜索加载（Tool Search）

Spring AI 社区有一个 `ToolSearchToolCallAdvisor`，思路和本项目的 `LoadSkillTool` 一致——按需加载，减少 token。区别在于它按需加载的是 tool 的定义，不是 skill 的 prompt。当系统有 20+ 个 tool 时，全量 tool 定义发过去可能消耗 55000+ token。它只给 LLM 发一个搜索工具，LLM 需要时先搜工具名和描述（支持语义向量/Lucene/正则三种搜索），匹配到的才展开定义发过去。实测省 34-64% token。

当前项目只有两个 tool，不急。但这是加更多 tool 时不炸 token 预算的关键手段。Maven 坐标 `org.springaicommunity:tool-search-tool`。

探索点：语义搜索 vs 关键词搜索的选择时机、搜到的 tool 定义缓存复用、和 `LoadSkillTool` 的协作（先搜 tool 还是先搜 skill？）。

### Prompt Caching

DeepSeek 和 OpenAI 都支持自动缓存 system prompt。第一个请求全量发送，后续请求命中缓存的 prefix 部分不再计费。当前 system prompt 近万字符、每次原样重发——如果利用缓存，token 成本可显著降低。

Spring AI 下的落地方式：在 `ChatOptions` 或 `DeepSeekChatOptions` 中寻找缓存相关配置，或直接在 `createRequest()` 里注入缓存策略标记。实现后通过日志对比缓存命中前后的 token 用量。

探索点：缓存什么时候命中、什么时候失效（一条消息变化就全量重发？）、流式和非流式行为是否一致。

### Token 用量统计与成本可视化

当前 `CharBasedTokenEstimator` 只做估算，不记录实际值。DeepSeek API 响应里有 `usage: {prompt_tokens, completion_tokens, total_tokens}`——Spring AI 的 `ChatResponse` 已经拿到了，只是项目没有存。

落地方式：在 `ChatHistoryAdvisor.after()` 中从 `ChatResponse.getMetadata()` 里取出 `Usage`，写入 `ai_chat_messages` 的预留字段或 `ai_chat_sessions` 的单独列。前端在会话详情或管理面板中展示每轮对话的 token 消耗。

探索点：不同提示词版本对 token 消耗的影响、思考模式 vs 非思考模式的 token 差异。

### 思考模式（Thinking）控制

DeepSeek V4 的 `thinking: {type: "enabled"}` 控制推理深度。当前项目始终开启（`application-ai.yml` 中配置），但简单任务不需要。比如"把这张图放大到 800x600"——关掉思考，响应快几秒。

Spring AI 下的落地方式：在 `DeepSeekChatOptions` 中设置 `thinking` 参数，然后根据 `site_config` 或请求参数动态调整。流式场景下 reasoning_content 为空时前端的折叠面板自然隐藏。

探索点：思考模式对工具调用的影响（有推理时 Edge 路由是否更准确？）、关思考后 response 中 `reasoning_content` 字段是 null 还是空字符串。

## 第二梯队：需要设计，但和已完成部分一脉相承

这些不是加配置就好的——需要写新代码，但底层接口已备好。

### 上下文结构分隔设计

来自 `5-19/p1-separator-design.md`。当前 system prompt 和 history 是无结构拼接的——一段文本接另一段。但 LLM 对不同位置的注意力权重不同：开头（Primacy）和结尾（Recency）权重高，中间段（Lost in the Middle）容易被忽略。应该在上下文结构中利用这一点——用 `## System`、`## History`、`## Current Task` 等标记把上下文分区，关键指令放首尾，历史放中间。工具结果用 `<tool>` 标签包裹和正文区分。

Spring AI 下的落地方式：在 `ChatHistoryAdvisor.before()` 的消息组装时注入结构标记，在 `PromptManager.buildSystemPrompt()` 中按规则排布。ChatClient 已经支持 `Prompt` 的多 message 结构，可以在 message 层面上分区。

探索点：XML 和 Markdown 两种分隔格式哪种 LLM 更容易遵循、分隔标记本身是否占用额外的 token 会被 LLM 当作指令、不同层次分隔的对比实验。

### 对话摘要压缩

当前 `SlidingWindowStrategy` 直接丢弃旧消息。摘要压缩的思路是把要被丢的旧消息先送给一个小模型，生成一段简短摘要，摘要保留，原文丢弃。同样 4000 token 预算，摘要版能覆盖更长历史。

Spring AI 下的落地方式：`CompressionStrategy` 接口已有 `compress()` 和 `tryCompress()`——新增一个 `SummaryCompressionStrategy` 实现即可。但摘要生成本身需要一次 LLM 调用（异步，不阻塞当前对话），这块需要独立设计。`LoadingStrategy` 接口的 `load()` 方法可以改为"加载最后 N 条完整 + 其余摘要"。

探索点：摘要质量对后续对话的影响、摘要生成的成本（用便宜模型 vs 用主模型）、摘要存储（存在消息表里还是单独的表）、摘要和滑动窗口可以叠加吗。

### RAG 检索增强

`5-20/p3-rag-design.md` 已有完整设计：知识库表、分块策略、BM25 + 向量混合检索、查询改写、重排序。当前项目用不到——没有大量文档需要检索，AI 不回答知识性问题。但如果目的是学习 RAG 怎么在 Spring AI 下跑通，这个项目刚好有素材（CLAUDE.md、设计文档、API 注释都可以入库）。

Spring AI 下有 `spring-ai-pgvector`、`spring-ai-elasticsearch` 等向量存储 starter，但不一定要用。MySQL 也能做——embedding 存 JSON 列，Java 里算 cosine similarity。规模小了比引入新组件更可控。

探索点：分块策略从简单（固定长度）到复杂（语义分块、标题分块、小到大），每一档对检索结果的影响；BM25 和向量的加权融合系数调参。

### 工具调用结果去重

来自 `5-19/p2-dedup-design.md`。工具调用循环中同一工具可能被多次调用，返回相似结果，白白占用窗口。实现一个 `DedupStrategy`（也是 `CompressionStrategy` 接口），检测历史中重复的 tool-call/tool-result 对，只保留最新那条。换策略只改 AiConfig 一行。

探索点：去重粒度（完全相同 vs 语义相似）、和滑动窗口的叠加（先去重再裁旧）、去重后的上下文连贯性。

### 上下文多维度评分

来自 `5-19/p2-freshness-scoring-design.md`。当前压缩只看位置——最新的保留，最旧的丢掉。但"最新"不等于"最重要"：一条关键的系统指令可能比 5 条闲聊更有保留价值。给每条消息按时间、角色、内容长度加权打分，压缩时丢掉最低分而非最旧的。同样是 `CompressionStrategy` 接口。

探索点：权重怎么分配（时间衰减函数、角色权重、内容长度阈值）、和滑动窗口的混合策略。

### 上下文健康检查

来自 `5-19/p2-health-check-design.md`。压缩之前先做质量检查：有没有"改过颜色"+"恢复默认"这种互相抵消的操作，有没有时间过期的引用，有没有矛盾的事实陈述。检查出问题后要么标记、要么修复、要么去掉——不让垃圾进 LLM。可以和 `CompressionAdvisor` 串联，也可以是独立的 `HealthCheckAdvisor`。

探索点：LLM 用作健康检查器（让它判断是否矛盾）vs 规则引擎（正则 + 关键词），异步检查 vs 同步阻塞。

### 上下文质量评估

上下文工程做好之后，你改一行 system prompt、调一下压缩窗口，怎么知道"改好了"？靠感觉不行。

落地方式：建一组测试问题，写死预期工具调用或预期回答。每次改完跑一遍，记录 token 用量、压缩触发次数、工具调用成功率、回答中引用来源的准确率。不需要复杂——一个 shell 脚本 + `curl` 调用 chat 接口就行。

探索点：评估和 A/B 测试在 Spring AI 下的可测试性——ChatClient 的 advisor 链能否用 mock 替换、ChatModel 能否用录制的 API 响应做 deterministic 测试。

## 第三梯队：框架层面，但是下一阶段的事

这些做早了是过度设计，但值得知道方向。

### Agent 抽象

当前只有一个 Agent（draw），`DrawService` 做了编排。等加第二个 Agent 时，提取 `AgentExecutor` 骨架——构建 prompt、注册 tools、调 ChatClient、处理 chunk——这些应该复用。Spring AI 的 `ChatClient` 已经是这个方向的一步，Agent 抽象是它的上层包装。

### 多 Agent 协作（P7）

规划-执行分离：主 Agent 做规划（"画一个登录流程图"），子 Agent 携带精简上下文执行（实际生成 XML）。Spring AI 没有原生的多 Agent 框架，需要自己调度。技术看点在于上下文隔离——两个 Agent 看不到对方的对话历史，只通过标准的 `AssistantMessage.ToolCall` 交换信息。

### 用户长期记忆（P4）

从多轮对话中提取用户偏好（"我喜欢极简风格"、"我是后端开发者"），存入记忆表，新对话开始时注入相关的记忆片段。技术看点在于：记忆提取的触发时机（对话空闲时异步？）、记忆的冲突消解（"我现在不喜欢极简了"）、记忆的检索方式（向量匹配还是关键词匹配）。

### 模型无关性

当前适配层只为 DeepSeek 写了 4 个文件。如果未来要接 Claude（等 Spring AI 官方支持成熟后），需要重新评估 `ModelMessageConverter` 策略模式的适配范围——Message 层、Options 层、Tool 定义层都需要翻译。这本身是一个有价值的学习课题。

## 如何选

如果把上面每个方向都做一遍，重心会从"博客系统"偏向"AI 实验室"——这取决于你更想用这个项目做什么。

当前最自然的节奏是：把第一梯队的 3 个主题作为"上下文工程实战"的延伸实验，不用改架构，收益即时可见。第二梯队选摘要压缩或 RAG 其中一个深入——每个都足够写一篇技术文章级别的文档。第三梯队保持关注，等 Spring AI 版本演进带来新的 API 能力。

每个主题都值得一篇独立的 `5-xx/` 设计文档 + 代码实验记录。
