# 06 — 模型适配层：DeepSeek 流式工具调用

## 解决了什么问题

Spring AI 的 `ToolCallAdvisor` 是通用的——它处理工具调用循环，但不知道 DeepSeek 模型的特殊性：**reasoning_content（推理过程）在流式场景下会丢失。**

具体来说：
1. DeepSeek V4 开启思考模式后，每个流式 chunk 带一小段 `reasoning_content`（"用户的输入是中文，我应该..."），聚合后是完整的推理过程
2. 工具调用场景下，LLM 返回 assistant(tool_call) → 后端执行工具 → 递归调用 LLM → 最终回复。递归时**对话历史里必须带着 reasoning_content**，否则 DeepSeek API 在非首轮调用时收到含 assistant(tool_call) 但无 reasoning 的消息会报 400
3. Spring AI 的原生 `ToolCallAdvisor` 把 `AssistantMessage` 原样放回历史，但 `DeepSeekAssistantMessage` 的 `reasoningContent` 字段在 Spring AI 核心类中不存在

这个 advisor 做一件事：**在流式工具调用循环中，捕获 reasoning_content 并注入回对话历史，让 DeepSeek 的思考模式正常工作。**

---

## 核心流程

```
用户请求
  → ChatHistoryAdvisor.before()
  → DeepSeekToolCallAdvisor.adviseStream()
      │
      ├─ 放行流式请求到 LLM
      │     │
      │     └─ 每个 chunk 到达时:
      │          ├─ 实时透传给前端（用户看到打字效果）
      │          └─ 从 DeepSeekAssistantMessage 中提取 reasoningContent
      │               → 追加到 ReasoningHolder
      │
      ├─ 流结束后: 聚合所有 chunk → 检查是否有 tool call
      │     │
      │     ├─ 无 tool call → 流结束
      │     │
      │     └─ 有 tool call:
      │          ├─ 执行工具（display_diagram / loadSkill）
      │          ├─ 拿到 ReasoningHolder 中完整 reasoning
      │          ├─ patchHistory(): 把 reasoning 注入 assistant(tool_call) 消息
      │          └─ reasoningLoop() 递归 → 再次请求 LLM
      │
      └─ 前端收到 text-delta → tool-call → ... → done
```

## 关键技术点

### 1. 多播分流（publish multicast）

```java
return captured.publish(shared -> {
    // 分支1: 实时流式输出 → 前端逐字显示
    Flux<ChatClientResponse> streaming =
            new ChatClientMessageAggregator()
                    .aggregateChatClientResponse(shared, aggregatedRef::set);

    // 分支2: 流结束后检查是否要递归处理工具调用
    Flux<ChatClientResponse> recursion = Flux.defer(() -> {
        // aggregatedRef 里是聚合完的完整响应
        // 有 tool call → 执行 → reasoningLoop() 递归
    });

    return streaming.concatWith(recursion);
});
```

一个流分两路走：一路实时发给前端（streaming），一路等流结束检查工具调用（recursion）。`concatWith` 确保先输出流式内容再处理递归。

### 2. Reasoning 捕获与回注

```
Chunk 1: { content: "好的，我来", reasoning: "用户想画图" }
Chunk 2: { content: "画一个流程图", reasoning: "需要横排布局" }
Chunk N: { content: "...", tool_calls: [{name:"display_diagram",...}] }

聚合后:
  reasoning = "用户想画图需要横排布局"
  content = "好的，我来画一个流程图"
  tool_calls = [display_diagram]

patchHistory 把 conversationHistory 中的 AssistantMessage(tool_call)
替换为 DeepSeekAssistantMessage(content + reasoning + tool_calls)
```

如果不做这一步——conversationHistory 里 assistant 消息没有 reasoning ——下一轮 DeepSeek API 调用会返回 400。

### 3. 递归循环

```java
private Flux<ChatClientResponse> reasoningLoop(..., List<Message> instructions) {
    // 1. 用当前 instructions（含历史 + 最新 tool 结果）发起请求
    // 2. 捕获 reasoning + 聚合
    // 3. 有 tool call → 执行工具 → patchHistory → 递归 reasoningLoop
    // 4. 无 tool call → 结束
}
```

每次递归用 `streamAdvisorChain.copy(this)` 生成新的 advisor 链副本，避免状态污染。递归最多持续到没有工具调用为止。

## Spring AI 架构分析

### OpenAI Chat Completion 规范

2023 年，OpenAI 发布了 Chat Completion API。请求是一个 JSON，包含 `model`、`messages` 数组、`temperature`、`max_tokens` 等参数。`messages` 里每条消息有 `role`（system/user/assistant/tool）和 `content`。响应带回 `choices[]`，每个 choice 里有 `message`。流式走 SSE，以 `data: [DONE]` 结束。

工具调用不是藏在 `content` 里靠提示词引导的——API 层面直接支持。请求里有一个独立的 `tools` 字段列出可用工具的定义，响应里 `message.tool_calls` 是另一个独立字段，直接以 JSON 数组返回模型决定调哪些工具：

```json
// 请求 — tools 和 messages 是两个独立的兄弟字段
{"model": "deepseek-chat", "messages": [...], "tools": [{"type":"function","function":{...}}]}

// 响应 — tool_calls 是 message 下的独立数组，不需要从 content 里 regex 拆
{"choices": [{"message": {"role": "assistant", "content": null, "tool_calls": [
    {"id": "xxx", "type": "function", "function": {"name": "display_diagram", "arguments": "{...}"}}
]}}]}
```

有 tool_calls 时 `content` 为 null，普通回复时 `tool_calls` 为空。ChatModel 要做的是把 JSON 里的 `tool_calls` 数组反序列化到 `AssistantMessage.ToolCall` ——框架的 `ToolCallAdvisor` 看到 `hasToolCalls() == true` 就知道该执行工具，完全不需要语义推断。

这个格式迅速成为了行业默认。后来入场的厂商基本都照此实现：DeepSeek、xAI Grok、Mistral、Kimi、Groq、Together AI、Fireworks、Perplexity，甚至本地推理项目 Ollama 和 vLLM 都内置了 OpenAI 兼容端点。DeepSeek 走得最远——除了响应 message 里多了 `reasoning_content`、请求支持 `thinking: {type: enabled}` 之外，和 OpenAI 别无二致。没有走这条路的主要是 Anthropic 的 Claude 和 Google Gemini 的原生 API，但 Spring AI 不是不能用它们——只是 ChatModel 内部把标准格式翻译到目标格式的工作量更大而已，对使用者来说仍是 `ChatClient.prompt().call()`，无感。

### Spring AI 的三层架构

顶层是你写业务代码的地方：`ChatClient.prompt().system("你是...").messages(userMsg).tools(toolList).call()`。ChatClient 收集这些调用，组装成 `Prompt`（`List<Message>` + `ChatOptions`），传给 Advisor 链。

中间是 Advisor 链：before → next → after。操作 `List<Message>`，不关心请求最终发到哪个模型。截断历史、注入上下文、触发压缩——都在这里。

底层是 `ChatModel` 接口，只有三行契约：`call(Prompt)`、`stream(Prompt)`、`getDefaultOptions()`。ChatModel 只负责格式翻译——把 `List<Message>`（Spring AI 标准格式）变成目标 API 的 HTTP JSON，把 HTTP 响应变回 `ChatResponse`（Spring AI 标准格式）。工具调用的识别和循环完全是框架的事——`ToolCallAdvisor` 识别到 `AssistantMessage.hasToolCalls()` 就调 `ToolCallingManager` 执行工具、把结果拼回 conversation history、再递归调 ChatModel。ChatModel 不用管这些。

### 问题出在哪：Message 的"通用语言"假象

`Message` 贯穿三层。但它的结构是按 OpenAI 规范定的：`AssistantMessage` 只有 `content` 和 `toolCalls`。DeepSeek 多出来的 `reasoningContent` 没地方放。Spring AI 给了 DeepSeek 一个子类 `DeepSeekAssistantMessage` 加了这个字段，`DeepSeekChatModel` 从 HTTP 响应取到 `reasoning_content` 后会塞进去。问题在于 Advisor 层不认识子类：`ToolCallAdvisor` 内部做消息拼接时只把 assistant 当 `AssistantMessage` 处理，`reasoningContent` 丢了。工具调用循环里下一轮 LLM 请求带着没 reasoning 的 assistant(tool_call) 消息，DeepSeek API 直接报 400。同理，`ChatOptions` 接口全是 OpenAI 的参数名（`temperature`、`maxTokens`……），DeepSeek 的 `thinking: {type: enabled}` 目前（Spring AI 1.1.5）无法通过通用配置路径设置。

所以本项目的 4 个适配文件不是在适配 DeepSeek API——那层官方做了。它们在补 Advisor 层和持久化层的洞，让框架不认识的那些字段不被丢弃。

### 要自己适配一个新模型，需要做什么

取决于目标模型的 API 偏离 OpenAI 格式多远。

**模型走 OpenAI 格式（最常见）。** 零代码。复用官方的 `OpenAiChatModel`，改 baseUrl 和 apiKey 即可。大部分国产模型和新模型都是这样接的，不需要看任何 Spring AI 源码。

**OpenAI 格式但多了几个特有字段。** 继承官方的 ChatModel，覆写一个方法。以本项目为例：`DeepSeekReasoningChatModel extends DeepSeekChatModel`，只覆写 `createRequest()` 把 `reasoningContent` 序列化到 JSON 里。看一眼官方 ChatModel 的 `createRequest` 实现作为模板就够了，不需要理解整个框架。

**模型 API 完全不同（Claude 原生、Gemini 原生）。** 需要完整实现 `ChatModel` 接口。核心工作只有两条翻译链：把 `List<Message>` 翻译成目标 API 的请求 JSON，把响应 JSON 翻译回 `ChatResponse`。工具调用方面只做格式翻译（`ToolDefinition` ↔ 目标 API 的工具定义格式、响应 `tool_calls` ↔ `AssistantMessage.ToolCall`），不负责工具执行的循环。可以参照 `OpenAiChatModel` 看翻译链怎么串，Spring AI 的基类 `AbstractToolCallSupport` 会联动 `ToolCallingManager`，不需要自己写工具调用循环。看一个官方实现的代码量就够了，不需要通读 Spring AI。

这 4 个文件等一件事：Spring AI 把 `AssistantMessage` 扩展出推理相关的字段。加了之后——全部可以删掉，缩成一行配置。

## 为什么 extends ToolCallAdvisor

Spring AI 的 `ToolCallAdvisor` 已经处理了大部分工具调用逻辑——初始化 tool context、执行工具、拼接 conversation history。重写它太复杂。

`DeepSeekToolCallAdvisor` 只覆写 `adviseStream()`——在保留原有工具调用逻辑的前提下，夹带了两个 DeepSeek 特有操作：
1. 从流式 chunk 中捕获 reasoningContent
2. 在执行完工具后将 reasoning 注入 conversation history

## Spring AI 升级后的清理清单

升级 Spring AI 版本后需检查：

| 检查项 | 条件 | 动作 |
|--------|------|------|
| `DeepSeekReasoningChatModel` 是否需要 | 官方 `DeepSeekChatModel.createRequest()` 是否已传递 reasoningContent | 可删除 |
| `DeepSeekToolCallAdvisor` 是否需要 | 官方 `ToolCallAdvisor` 是否支持 `DeepSeekAssistantMessage` | 可简化 |
| `AssistantMessage` 是否有 reasoningContent | 官方是否加了这个字段 | 去掉 `DeepSeekAssistantMessage` 的 workaround |

## 总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 正确性 | ⭐⭐⭐⭐ | 解决了 reason 丢失导致的 400 错误 |
| 复杂度 | ⭐⭐ | Reactor 多播 + 递归 + patchHistory 三者交织，调试困难 |
| 可维护性 | ⭐⭐ | 等 Spring AI 修复后这个类应该删掉 |
| 可观测性 | ⭐ | reasoning 捕获和 patch 只有日志，无可视化 |

> 一句话：这是 Spring AI + DeepSeek 的临时补丁，不是永久设计。目标是等 Spring AI 修复后整体删除。
