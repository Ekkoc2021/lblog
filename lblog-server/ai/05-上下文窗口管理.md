# 05 — 上下文窗口管理（P2）

## 解决了什么问题

对话越来越长，历史消息不断堆积。每次 LLM 调用都带着从第一轮到现在的所有消息，token 消耗持续膨胀。长到一定程度后：
- 旧消息的注意力权重极低，但仍占窗口
- 有效信息被稀释，LLM 容易忽略中间段关键内容
- 工具调用 XML 可能一次就占几千 token，直接挤爆预算

P2 做一件事：**每次 LLM 调用前检查上下文大小，超了就从最旧的消息开始裁。**

### 怎么用

在 `AiConfig.drawChatClient()` 中创建策略实例，同时传给 `ChatHistoryAdvisor`（加载面）和 `CompressionAdvisor`（压缩面），同一个对象两端生效：

```java
SlidingWindowStrategy slidingWindow = new SlidingWindowStrategy(20, 30);

.defaultAdvisors(
    new ChatHistoryAdvisor(..., slidingWindow, ...),  // 加载时只取 20 条
    new DeepSeekToolCallAdvisor(...),
    new CompressionAdvisor(estimator, slidingWindow, maxTokens, ...))  // 运行时兜底
```

换策略只改这一处——比如换成 `new SummaryCompressionStrategy(10)`，加载和压缩行为自动对齐。Token 预算通过 `application.yml` 的 `ai.context.max-history-tokens` 配置，由 `CharBasedTokenEstimator`（字符数 / 2）做快速估算。

---

## 架构

```
ChatHistoryAdvisor (加载时已按 LoadingStrategy 截断，只取最后 20 条)
        │
        ▼
DeepSeekToolCallAdvisor (工具调用循环 — 可能新增 display_diagram XML 大消息)
        │
        ▼
CompressionAdvisor (最后一道防线)
  ├─ token 没超 → 问 strategy.shouldCompress() → 超过 30 条则裁到 20 条
  ├─ token 超了 → 直接压缩 → 循环 tryCompress 直到预算内
  └─ 保护：system 消息永不被裁
```

**两个压缩触发条件：**
- **条数触发** — 消息超过 30 条（`minTrigger`），不管 token 多少
- **Token 触发** — 总 token 估算超过 `maxHistoryTokens`（默认 4000）

---

## 核心流程

### CompressionAdvisor.compressIfNeeded()

```java
// 1. 分离 system 消息（永不压缩）
Message system = messages.getFirst();
List<Message> target = messages.subList(1, messages.size());

// 2. 策略压缩
List<Message> compressed = strategy.compress(target);

// 3. 补回 system
List<Message> result = new ArrayList<>();
result.add(system);
result.addAll(compressed);

// 4. token 还超 → 循环调 tryCompress，一次丢一条最旧的
int guard = 50;
while (result.size() > 1 && estimateTokens(result) > maxHistoryTokens && guard-- > 0) {
    List<Message> after = strategy.tryCompress(result.subList(1, result.size()));
    if (after.isEmpty() || after.size() >= result.size() - 1) break;
    result = new ArrayList<>();
    result.add(system);
    result.addAll(after);
}
```

### SlidingWindowStrategy 的行为

| 方法 | 触发条件 | 行为 |
|------|---------|------|
| `load()` | 加载阶段 | 从 DB 取最后 20 条 |
| `shouldCompress()` | 消息数 > 30 | true |
| `compress()` | token 超 / shouldCompress | 保留最后 20 条 |
| `tryCompress()` | 仍超预算 | 丢掉最旧的一条 |

## 关键设计评审

---

### 决策 1：条数触发 + Token 触发双保险

**当前做法：** `shouldCompress` 按消息数判断（> 30 条），token 超预算直接压。两类条件独立工作。

**为什么：** 消息数触发防的是"长对话慢慢膨胀"（30 条 × 每条约 100 token = 3000 token，还安全）。Token 触发防的是"单条消息炸窗口"（display_diagram XML 一次 2000+ token，虽然只有 3 条消息但也可能超预算）。

**替代方案：**

| 方案 | 做法 | 问题 |
|------|------|------|
| A. 双条件（当前） | 条数 or token 任一触发 | — |
| B. 纯 token | 只按 token 判断 | 条数多但 token 不超时不压缩，可能没必要 |
| C. 纯条数 | 只按消息数判断 | 单条大 XML 可能绕过 |

**评价：** A 是最安全的选择。条数触发是快速路径（不计算 token），token 触发是兜底。

---

### 决策 2：system 消息保护

**当前做法：** CompressionAdvisor 把 system 消息从消息列表中剥离，只压缩非 system 部分，压缩完再补回。

**为什么：** system prompt 包含了 AI 的核心行为约束（绘图规则、XML 规范、Edge 路由规则等），删掉它 AI 就不知道该怎么画图了。这是不可裁的部分。

**评价：** 正确。所有压缩操作的保护对象都是 system 消息。

---

### 决策 3：tryCompress 的循环 + guard

**当前做法：** `while (仍超预算 && size > 1 && guard-- > 0)` 循环调用 `tryCompress`，每次丢掉最旧的一条。

**为什么用 guard：** 某些策略可能返回"size 不变"的列表（如摘要策略替换了一条为摘要但没减少条数），需要 guard 防止死循环。

**之前的问题：** 用 `result.size() < lastSize` 判断是否有压缩空间，但某些策略可能减少了 token 而不减少条数，导致循环提前退出。改为 guard 计数器更安全。

**评价：** guard=50 在这个项目规模下够用——一次对话最多几十条消息。

---

### 决策 4：CompressionAdvisor 的职责边界

**当前做法（迭代后）：**
- CompressionAdvisor：拆分/补回 system、token 估算、循环控制
- CompressionStrategy：只管"怎么裁切"（compress / tryCompress）
- LoadingStrategy：只管"加载时取什么"（load）

**之前的问题：** SlidingWindowStrategy 一度持有 TokenEstimator，感知 token 预算。重构后策略不碰 token，职责完全上收到 CompressionAdvisor。

**评价：** 这个重构方向正确。策略只管"裁什么"，上层管"裁不裁"——开闭原则的经典应用。

## 使用指南

### 基本配置

所有窗口参数在 `AiConfig.drawChatClient()` 中设置：

```java
SlidingWindowStrategy slidingWindow = new SlidingWindowStrategy(20, 30);
//                                              maxMessages=20 — 压缩到 20 条
//                                              minTrigger=30 — 超过 30 条触发

new CompressionAdvisor(tokenEstimator, slidingWindow, maxHistoryTokens, order);
//                                    maxHistoryTokens — 来自 application.yml
```

`application.yml`：

```yaml
ai:
  context:
    max-history-tokens: 4000   # 历史消息 token 预算
```

### 调整策略

**对话长度变长 → 加大窗口：**

```java
// 保留最后 40 条，超过 50 条才触发压缩
new SlidingWindowStrategy(40, 50);
```

**想省 token → 缩小窗口或降低预算：**

```java
// 只保留 10 条
new SlidingWindowStrategy(10, 20);

// 或降低 token 预算
// ai.context.max-history-tokens: 2000
```

### 换一种压缩策略

实现 `CompressionStrategy` 接口，替换 AiConfig 中的策略实例即可：

```java
// 假设未来有摘要压缩
new SummaryCompressionStrategy(keepFull=10);  // 保留最后 10 条完整，其余摘要

// 或什么也不压缩
new NoOpCompressionStrategy();
```

三个 advisor 不需要改动，`ChatHistoryAdvisor.load()` 和 `CompressionAdvisor.compressIfNeeded()` 都自动适配新策略。

### 组合关系一览

```
AiConfig.drawChatClient()
  │
  ├─ new SlidingWindowStrategy(20, 30)        ← 策略实例
  │
  ├─ new ChatHistoryAdvisor(..., slidingWindow) ← 加载面：load() 只取 20 条
  │
  └─ new CompressionAdvisor(..., slidingWindow) ← 压缩面：兜底工具结果膨胀
```

同一个 `slidingWindow` 对象传给两个 advisor，加载和压缩的窗口参数一致。

## 实现层面发现的问题

### 🟡 Token 估算精度低

`CharBasedTokenEstimator` 用 `text.length() / 2 + 1`。对中文（一个字符 ≈ 1-2 token）和对英文（一个单词 ≈ 1-2 token）误差差异大。

**影响：** 中文消息的 token 预估可能偏高，导致提前触发压缩；全英文消息预估偏低，可能实际已超出预算。当前没有安全边际系数。

**建议：** 引入 jtokkit 精确估算，或在现有估算上乘以 1.2 安全系数。

---

### 🟢 小问题速览

| 问题 | 说明 |
|------|------|
| SlidingWindowStrategy.load() 仍走全量查询 | 从 DB 取出全量再内存截取，未用 loadRecentMessages(N) |
| maxHistoryTokens 硬编码默认值 | `@Value("${ai.context.max-history-tokens:4000}")` 兜底 4000，和 CompressionAdvisor 紧耦合 |
| CompressAdvisor 无指标 | 不知道压缩了多少次、省了多少 token |
| 无 SummaryStrategy | 当前只有滑动窗口一种策略，旧消息直接丢弃而不是摘要保留 |

## 总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 职责分离 | ⭐⭐⭐⭐⭐ | system 保护、循环控制、策略裁切三者解耦清晰 |
| 健壮性 | ⭐⭐⭐⭐ | 双触发条件 + guard 防死循环 |
| 扩展性 | ⭐⭐⭐⭐ | 策略模式，换压缩算法只改一个类 |
| 精度 | ⭐⭐ | Token 估算简陋 |
| 可观测性 | ⭐ | 无指标，不知道压缩效率 |

> 一句话：窗口管理是最成熟的一个子系统——从架构到细节经过多轮重构沉淀。剩余债务是 token 估算精度和可观测性。
