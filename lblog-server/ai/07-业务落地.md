# 07 — 业务落地：从基础设施到 Draw Agent

## 通用基础设施

### 流式方案

AI 调用是慢操作。DeepSeek 思考加生成一个完整绘图回复通常需要几秒到十几秒。普通 HTTP 请求-响应模式下用户盯白屏等——体验不可接受。必须用流式。

流式能跑通，前提是大模型 API 本身支持。DeepSeek 的 Chat Completion API 和 OpenAI 一样，请求里传 `stream: true`，API 就不再一次性返回完整响应，而是以 SSE 格式持续推送小块数据。每个 chunk 是一个 JSON，里面是 `choices[0].delta`（增量）而非 `choices[0].message`（全量）：

```
data: {"choices":[{"delta":{"role":"assistant"},"index":0}]}
data: {"choices":[{"delta":{"content":"好的"},"index":0}]}
data: {"choices":[{"delta":{"content":"，我来画"},"index":0}]}
data: [DONE]
```

这个"逐步推送"之所以成立，是因为 LLM 的生成原理是 autoregressive decoding——一个 token 一个 token 地预测，生成一个就能立刻推一个，不需要等全文完成。

Spring AI 的 `ChatModel.stream()` 返回 `Flux<ChatResponse>`，框架内部把 API 的 SSE chunk 逐个转成 `ChatResponse` 对象推给调用方。到此，流已经到后端内存里了。下一步是怎么传给浏览器。

API 返回的流到了后端手里，传给浏览器有两个选择。

SSE（Server-Sent Events）是 HTML5 标准的一部分。它不新建协议——就是 HTTP 长连接加一套极简文本格式。客户端发普通 GET 请求，带 `Accept: text/event-stream`。服务端保持连接打开，每条消息以 `data:` 开头、空行分隔，一直写。浏览器端用原生 `EventSource` 对象接收，断线自动重连无需写代码。因为走 HTTP，所有代理和负载均衡器天然兼容。Spring 将它封装成了 `SseEmitter`：Controller 返回 emitter 实例，业务代码拿到后 `send()` 发事件，`complete()` 收尾。

WebSocket 是另一条路。它不是 HTTP 扩展——是独立协议。握手阶段借道 HTTP 发一个 Upgrade 请求，服务端回 101 Switching Protocols，TCP 连接升级为 WebSocket。之后双方可以同时、独立地在同一条连接上互相发送文本帧或二进制帧，每帧元数据开销仅 2-14 字节。代价是断线重连需要自己写、部分代理需要显式配置 Upgrade 透传、空闲连接可能需要自己发 ping/pong 维持。

| | SSE | WebSocket |
|------|------|------|
| 通信方向 | 服务端 → 浏览器 | 服务端 ↔ 浏览器 |
| 协议层 | HTTP 长连接 | WebSocket（TCP 升级） |
| 浏览器 API | `EventSource` | `WebSocket` |
| 断线重连 | 浏览器原生支持 | 需手动实现 |
| 代理兼容 | HTTP 天然通过 | 需配置 Upgrade 透传 |
| 适合场景 | 流式输出、订阅、通知 | 聊天、协作编辑、游戏 |

AI 对话的通信模式是单向推流。浏览器通过普通 POST 发用户消息给后端，后端流式推回 AI 回复，推完连接关闭。这期间浏览器不需要发任何东西。新消息是下一次独立的 POST。这个模式落在 SSE 最强的那一档——服务端推、长时运行、需要流式、不需要双向。WebSocket 多出来的双向能力在这里是负担，不是优势。

整个链路是两层 SSE 串联：DeepSeek API 的 SSE → Spring AI `Flux<ChatResponse>` → `SseEmitter` → 浏览器。

### 长时任务与请求线程解耦

#### 问题：不做任何处理会发生什么

一次 AI 绘图调用从发请求到收到完整回复，快则 5 秒，慢则 30 秒以上（思考模式 + 工具调用 + 多轮递归）。如果不做任何处理，这段逻辑直接跑在 Tomcat 的请求线程上。

Tomcat 默认有 200 条请求线程。每条线程被一个 AI 调用占住 10-30 秒。这条线程在此期间什么事都不做——它在阻塞等待 DeepSeek API 返回、等工具执行完成。用户的浏览器在等，线程也在等，但线程本可以去处理另一个请求。

问题不是"某一天突然耗尽"，而是随着并发上升，三个阶段递进恶化。

**轻度：响应变慢。** 10 个 AI 请求各占一条线程，空闲线程减少。新进来的普通请求——查会话列表、加载配置——这些本该毫秒级响应的操作开始排队等线程。延迟从 5ms 变成 500ms。用户感知不到"AI 慢"，但会感知"整个站变慢了"。

**中度：非 AI 接口被拖垮。** AI 请求和登录、查配置、查会话共享同一个 Tomcat 线程池。当大部分线程被 AI 请求占住时，这些和 AI 完全无关的接口也排不上号。根因在 AI 接口吃光了线程，但用户看到的现象是登录转圈、配置加载卡住——"整个站都坏了"。

**重度：拒绝连接。** 线程池全满，请求队列也溢出了（默认 100），Tomcat 开始拒绝新的 TCP 连接。这时候不是慢，是打不开。恢复要等 AI 调用超时释放线程，但这期间用户可能已经刷新多次，每一次刷新又提交一个新请求，压得更死。

根本原因是两种截然不同的任务混在同一个线程池里。Tomcat 请求线程设计来处理"接收 → 处理 → 返回"的快速循环。AI 调用是"接收 → 等 20 秒 → 返回"。当这两种混在一起，长任务挤走了短任务的资源。

#### 解法

本质上是把"等待"从请求线程中拆出来。请求线程只负责接收请求、校验参数、创建执行上下文，然后立刻释放。"等 AI 返回"这件事交给一个专门的执行载体去做。

四种常见方案。

**方案 A：WebFlux 全链路响应式。** Spring WebFlux 是 Spring 的响应式 Web 框架，跑在 Netty 而非 Tomcat 上。它的核心假设是：从 Controller 到 Service 到数据库再到外部 API，整条链路上没有任何方法阻塞线程。Controller 返回的是 `Mono<T>` 或 `Flux<T>`（而非 `T` 或 `SseEmitter`）。当一个请求在等外部 API 返回时，线程不等着——它去处理别的请求。Netty 的 event loop 通常只有 CPU 核数那么多条线程，却能承载数千并发连接。

具体到 AI 流式场景：ChatModel 返回 `Flux<ChatResponse>`，Controller 直接把这个 Flux 转成 `Flux<ServerSentEvent>` 返回。Spring 框架在底层把每个元素写入 HTTP 响应。全程没有 `@Async`、没有 `SseEmitter`、没有额外线程池——因为不需要"等"，也就不需要"把等待交给别的线程"。

但对这个项目不适用，原因不在 WebFlux 本身，而在于项目已经建在 Spring MVC 上。MVC 和 WebFlux 不能简单地在一个端口共存——它们是两套不同的 HTTP 运行时。迁移意味着：

- 所有 Controller 从返回 `T`/`SseEmitter` 改为返回 `Mono<T>`/`Flux<T>`
- MyBatis 需要响应式替代（Spring Data R2DBC 或 MyBatis Reactive），因为 JDBC 是阻塞的——一个阻塞调用会卡住整个 event loop 线程
- Spring Security 配置方式完全不同（`ServerHttpSecurity` 替代 `HttpSecurity`）
- 过滤器、拦截器、异常处理器全部换成响应式版本
- 团队需要理解 Reactor 的订阅模型、backpressure、冷热 Flux、Schedulers——这些不是"学一个新 API"，是"换一种编程范式"

为了一项 AI 功能把整站技术栈从 MVC 换到 WebFlux，投入产出严重不成比例。如果项目一开始就建在 WebFlux 上，这会是最自然的方案。

**方案 B：Servlet 3.1 AsyncContext。** 不换框架，直接用 Servlet 规范的异步能力。写起来大致是这样：Controller 不返回 `SseEmitter`，而是拿到 `HttpServletRequest` 后手动调 `request.startAsync()` 获得一个 `AsyncContext`，然后把业务逻辑通过 `asyncContext.start(() -> { ... })` 提交给线程池。业务线程在回调里获取 `asyncContext.getResponse().getWriter()` 手动拼 SSE 格式写入数据，最后调 `asyncContext.complete()` 收尾。请求线程在 `startAsync()` 后立即返回容器。

这完全可行，而且本质上当前方案走的就是这条路——`SseEmitter` 底层就是调了 `request.startAsync()`。如果用 AsyncContext 原生 API，好处是不依赖 Spring 的 SSE 封装，可以精确控制超时、重连、错误处理的每一环。代价是需要自己处理 SSE 的文本格式化（每条消息前面加 `data:`、空行分隔、多行拼接），需要自己管理 `response.getWriter()` 的 flush 和线程安全，需要自己实现超时回调——这些事情 `SseEmitter` 都做了。

选 `SseEmitter` 而非裸 AsyncContext 的逻辑是：这一层没有定制需求。SSE 的事件格式就是标准的 `data: json\n\n`，超时逻辑就是"3 分钟到了 close 连接"，没有 WebSocket 那种自定义帧格式或心跳协议。在这些点上重新造轮子的回报为零。

**方案 C：平台线程池 + @Async。** 如果项目在 Java 17 上（没有虚拟线程），这就是最直接的方案。`@Async` 提交到 `ThreadPoolExecutor`，线程池配置 50-100 条平台线程。每条线程 2MB 栈空间，100 条 = 200MB 栈内存。AI 场景下大部分时间在 IO 阻塞，栈空间利用率极低——线程在等 API 返回，但 2MB 栈占着。够用的时候够用：如果并发 AI 请求不到 20 个，20 条平台线程撑得住。但并发上去后栈内存 + 上下文切换逐渐成为问题。

**方案 D：虚拟线程 + @Async（当前方案）。** 保留 Spring MVC + `SseEmitter` + 同步代码风格，把 @Async 线程池的工厂从 `Thread.ofPlatform()` 换成 `Thread.ofVirtual()`。虚拟线程阻塞时不占 OS 线程栈（堆上分配、按需增长），JVM 可以在几个 OS 线程上调度成百上千个虚拟线程。代码仍然是同步的：`emitter.send()`、`chatResponse.getResult()`——不需要理解 Reactor 的 subscribe 和 backpressure。

| | 编程模型 | 线程开销 | 调试难度 | 适合场景 |
|------|---------|---------|---------|---------|
| WebFlux 全链路 | 响应式 | 极低 | 高 | 新项目，团队熟悉 Reactor |
| AsyncContext | 同步/异步混用 | 中 | 中 | 不想依赖 Spring 高层封装 |
| 平台线程池 | 同步 | 高（2MB/线程） | 低 | Java 17，低并发 |
| 虚拟线程（当前） | 同步 | 极低 | 低 | Java 21+，同步风格优先 |

当前选择方案 D 的逻辑：不想为了一项 AI 功能把整个项目从 MVC 切到 WebFlux，Java 21 也提供了比平台线程更好的并发模型。虚拟线程让 AI 调用代码保持同步的可读性和可调试性，同时不给 Tomcat 线程池造成压力。

#### 当前实现

一次 AI 请求涉及三条线程。

Tomcat 请求线程：DispatcherServlet → DrawController。校验开关、限流、创建会话、创建 SseEmitter。然后调用 `drawService.chatStream()`——由于方法标注了 `@Async`，Spring 不立即执行，而是把调用提交到虚拟线程池，Controller 线程立刻返回。整个过程毫秒级。Controller 返回 `SseEmitter` 给 Spring，Spring 接管 HTTP 长连接。请求线程回到 Tomcat 池。

@Async 虚拟线程：这就是 `chatStream()` 实际执行的地方。构建 system prompt → 调 ChatClient → 等 DeepSeek API 返回 → 收到 chunk 调 `emitter.send()` → 工具调用循环 → 直到流结束。这个线程的生命周期和一次 AI 调用的生命周期一致。

心跳线程：独立的 `ScheduledExecutorService`，单线程、守护模式、平台线程。每隔 5 秒往 emitter 发空事件。正常时瞬间返回。客户端断开后 `send()` 抛 IOException → catch 到后 `asyncThread.interrupt()` 中断虚拟线程 → 阻塞 IO 立即退出 → API 调用终止。为什么心跳不用虚拟线程？因为虚拟线程擅长"大量并发 IO"，不擅长"精确定时"——被挂起后恢复时间不确定。而 `ScheduledExecutorService` 就是为严格周期调度设计的。

线程池配置：

```java
new ThreadPoolExecutor(
    20, 20, 0L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    Thread.ofVirtual().factory()
);
```

20 个虚拟线程并行、100 容量的有界队列、`CallerRunsPolicy` 拒绝策略——满了由调用者线程自己跑，等于降级为同步，不丢请求。虚拟线程的 20 不是 20 个 OS 线程的开销，底层只占几个 carrier 线程。

三条线程通过 SseEmitter 协作。emitter 是唯一的共享点：虚拟线程往它写数据，心跳线程往它发心跳，Spring 从它读数据写 HTTP 响应。不需要锁——所有 send 调用都在同一虚拟线程上，心跳 send 和业务 send 之间有自然时序间隔。

### 断连检测

不管流式还是非流式，AI 调用都可能持续十几秒。用户可能中途关浏览器、切页面、断网。如果后端不感知到，API 调用继续跑完——白白消耗配额。这个问题和 SSE 无关：任何长连接方案（SSE、WebSocket、gRPC stream）或长时处理都要解决它。

本项目的解法是心跳加虚拟线程中断。DrawService 每次进入 AI 调用前，在独立的 `ScheduledExecutorService`（单线程、守护模式）上启动一个定时任务，每隔几秒往 emitter 发一个空事件 `{}`：

```java
ScheduledFuture<?> heartbeat = heartbeatScheduler.scheduleAtFixedRate(() -> {
    try {
        emitter.send(SseEmitter.event().data("{}"));
    } catch (Exception e) {
        asyncThread.interrupt();  // ← 关键：中断业务线程
        throw new RuntimeException(e);
    }
}, intervalSeconds, intervalSeconds, TimeUnit.SECONDS);
```

正常时 send 瞬间返回。客户端断开后 TCP 连接断开，`send()` 抛 IOException。catch 到异常后执行 `asyncThread.interrupt()`——`asyncThread` 是执行 AI 调用的虚拟线程。虚拟线程收到中断后，阻塞 IO 立即退出，Reactor 流式订阅终止，API 调用结束。

整个链路——从"浏览器关闭"到"API 调用终止"——不超过一个心跳间隔。

资源清理走两层：`emitter.onCompletion()` 回调在正常结束时取消心跳，`finally` 块在异常时兜底取消。心跳线程池配置了 `setRemoveOnCancelPolicy(true)`，cancel 后任务立即从队列移除，不等 GC。

注意低版本 Spring 的 `SseEmitter.send()` 有线程安全问题——并发 send 时内部状态可能错乱。当前实现中所有 send（心跳、reasoning、text-delta、done）都发生在同一虚拟线程内，不存在并发。如果未来改为多线程推送事件，需要先升级 Spring 或加同步保护。

### 速率限制

`DrawRateLimiter` 用 Caffeine 缓存实现 IP 级别限流。Caffeine 的 `expireAfterWrite(1, TimeUnit.MINUTES)` 自然形成滑动窗口效果——每个 IP 的计数器在写入后一分钟自动过期。每来一个请求，计数器加一。超了阈值就拒绝。限制值当前硬编码在 `MAX_REQUESTS_PER_MINUTE = 10`，后续应改为 `application.yml` 配置项。

这个限流器放在 Controller 层——在 `SseEmitter` 创建之前、AI 调用之前。被限流的请求连 emitter 都不会创建，直接返回错误。

## Draw Agent：装配车间

通用方案就绪，剩下的工作是把前面六篇拆开的零件，按 draw agent 这个具体场景装起来。

### 一次请求的完整链路

前端发起 `POST /api/v1/draw/chat`。DrawController 顺序做四件事：检查 AI 绘图开关（`site_config.ai_draw_chat_enabled`）、校验消息列表非空、IP 限流检查、如果没有 sessionId 就创建新会话。之后创建超时 3 分钟的 `SseEmitter`，丢给 DrawService，返回 emitter 给 Spring——Spring 接管 HTTP 长连接。

```
DrawService.chatStream (@Async 虚拟线程)
  ├─ 启动心跳
  ├─ buildFullSystemPrompt()
  │     ├─ DrawPromptManager: 角色约束 + XML 规范 + Edge 路由
  │     └─ SkillSystemPromptBuilder: 懒加载技能提示（一行）
  ├─ ChatClient.prompt()
  │     .system(prompt + xmlContext)
  │     .messages(userMessage)
  │     .tools(displayDiagramTool, loadSkillTool)
  │     .advisors(sessionId, modelName)
  │     .stream()
  │       → reasoning → SSE {"type":"reasoning","delta":"..."}
  │       → content   → SSE {"type":"text-delta","delta":"..."}
  │       → tool-call → DisplayDrawTool → SSE {"type":"tool-call",...}
  ├─ SSE {"type":"done","sessionId":"42"}
  └─ emitter.complete()
```

### SSE 事件模型

五种事件，各自有 type 字段让前端 switch：

| 事件 | 发出方 | 触发时机 | 前端行为 |
|------|--------|---------|---------|
| `heartbeat` | 心跳线程 | 定时 | 无操作 |
| `reasoning` | DrawService | DeepSeek 返回推理 chunk | 折叠面板展示 |
| `text-delta` | DrawService | LLM 返回文本 chunk | 聊天框逐字渲染 |
| `tool-call` | DisplayDrawTool | LLM 调 display_diagram | draw.io 编辑器渲染 |
| `done` | DrawService | 流结束 | 保存 sessionId |

### 装配清单

```
零件                        装配位置
─────────────────────────────────────────
DrawPromptManager            → 系统提示词（角色、XML 规范、Edge 路由）
SkillSystemPromptBuilder     → 懒加载技能提示（一行）
ChatClient (with advisors)   → LLM 调用骨架
  ├─ ChatHistoryAdvisor      → 加载历史 + 保存消息
  ├─ DeepSeekToolCallAdvisor → 工具调用循环
  └─ CompressionAdvisor      → token 兜底
DisplayDrawTool              → @Tool 注册
LoadSkillTool                → @Tool 注册
AgentStreamTransport         → 传输抽象（SSE 事件 + 心跳 + 断连）
虚拟线程池                   → @Async 异步执行
Caffeine 限流                → 入口防护
```

`@Async("diagramTaskExecutor")` 把整个流程放在 20 核心虚拟线程池上跑。

### 传输层抽象

`AgentStreamTransport` 不定义业务事件——只有一个 `send(eventType, data)` 方法。事件名和数据结构由 Agent 自行约定（draw 用 `"text-delta"`、`"reasoning"`、`"tool-call"`、`"done"`），传输层只负责把 data 发出去，不管里面是什么。

`SseStreamTransport` 只管 SSE 协议：把 data 序列化为 JSON，写 `data:` 前缀，管理 emitter 和心跳。`heartbeatIntervalSeconds <= 0` 时跳过心跳。未来 WebSocket 实现同一个接口即可——Agent 代码零改动。

### DisplayDrawTool

LLM 调它来生成图表。职责链：`MxCellValidator.validate(xml)` 校验 → `sanitize(xml)` 修复 → `wrapWithMxFile(xml)` 包装完整 mxfile 结构 → 通过 ToolContext 拿 `AgentStreamTransport` → `transport.send("tool-call", ...)` 发给前端。同时返回字符串给 LLM 告知结果。
