# 02 — 提示词管理

## 解决了什么问题

最初 `PromptManager.java` 里有约一万字符的系统提示词以 Java 15 文本块硬编码。每次调提示词要改代码→编译→部署，无法在生产环境快速试错。

提示词管理系统解决三件事：
1. **解耦** — 提示词从代码移到数据库，热修改即时生效
2. **审计** — 谁改了什么、什么时候改的，有日志可查
3. **版本** — 旧版本保留可回溯，改坏了能回滚

---

## 实现方式

### 加载优先级

```
AiPromptService.getPrompt("draw", "system-default")
  → 1. 查 Caffeine 缓存
       ├─ 命中 → 直接返回
       └─ 未命中 ↓
  → 2. 查 DB ai_prompts WHERE module=draw AND prompt_key=system-default AND is_active=1
       ├─ 有 → 缓存 + 返回
       └─ 没有 ↓
  → 3. 加载 classpath:prompts/draw/system-default.md
       ├─ 文件存在 → 返回内容（不写 DB）
       └─ 文件不存在 → 报错
```

注意：**文件不会自动写入 DB**。文件只在 `POST /api/v1/admin/ai/prompts/seed?module=draw` 调用时才导入 DB。

设计意图：开发时直接编辑 md 文件重启即生效，省去建表/写管理端的步骤。生产环境通过管理端写入 DB，秒级生效不需重启。

### 版本策略：INSERT-only

每次修改不是 UPDATE 旧记录，而是 INSERT 一条新记录（version + 1），旧记录 deactivate。效果：

- 所有历史版本都在表里，可查可回滚
- 审计日志记录了 old_content → new_content 的完整 diff
- 回滚：把旧版本的 is_active 设回 1，当前版本设 0

### 缓存策略

Caffeine 内存缓存，TTL=0（永不过期），上限 100 条。手动调 `POST /api/v1/admin/ai/prompts/reload` 清空缓存。

这样设计的好处：生产环境改了提示词，调一下 reload 接口立刻全量生效，不需要等服务发现/配置中心/重启。

## 关键代码路径

```
DiagramService.chatStream()
  → PromptManager.buildSystemPrompt(modelId, minimalStyle)
      → AiPromptService.getPrompt("draw", "system-default")    ← 核心指令
      → AiPromptService.getPrompt("draw", "system-extended")    ← 强力模型扩展
      → AiPromptService.getPrompt("draw", "style-minimal")      ← 风格指令
      → 组装：default + (extended if 强力模型) + (style if 极简模式)
      → 替换占位符 {{MODEL_NAME}} → 当前模型名
```

### 数据模型

```
ai_prompts 表
├── module         VARCHAR(50)   # 模块标识：draw / chat / codegen
├── prompt_key     VARCHAR(100)  # 提示词标识：system-default / style-normal
├── content        TEXT          # 提示词全文（Markdown）
├── version        INT           # 版本号递增
├── is_active      TINYINT(1)    # 是否生效
└── effective_from/to DATETIME   # 灰度发布（预留，未用到）

ai_prompt_audit 表
├── prompt_id      BIGINT        # 关联 ai_prompts.id
├── old_content    TEXT          # 修改前内容
├── new_content    TEXT          # 修改后内容
├── action         VARCHAR(20)   # CREATE / UPDATE / DEACTIVATE / SEED
└── operator       VARCHAR(100)  # 操作人
```

### 管理端 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/admin/ai/prompts | 列表（按 module 筛选） |
| GET | /api/v1/admin/ai/prompts/{id} | 详情 |
| GET | /api/v1/admin/ai/prompts/{id}/versions | 历史版本 |
| POST | /api/v1/admin/ai/prompts | 新增 |
| PUT | /api/v1/admin/ai/prompts/{id} | 更新（INSERT 新版本） |
| DELETE | /api/v1/admin/ai/prompts/{id} | 软删除 |
| POST | /api/v1/admin/ai/prompts/reload | 清除缓存 |
| POST | /api/v1/admin/ai/prompts/seed | 文件导入 DB |

## 设计决策评审

下面每个点遵循一样的结构：**当前做法 → 为什么是这个做法 → 有没有更好的做法**。

---

### 决策 1：DB 优先 + 文件兜底

**当前做法：** `AiPromptService.getPrompt()` → 缓存 → DB → classpath 文件 → 报错。文件只在 seed 时进 DB。

**为什么这样做：** 生产通过管理端热修改即时生效（秒级），开发直接编辑 md 文件重启即生效（省去建表步骤）。不引入配置中心等外部组件。

**替代方案：**

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 纯 DB（当前方向） | 文件只做 seed 的源，运行时全走 DB | 热修改、审计、版本一应俱全 | 开发期要 seed 才能用 |
| B. 纯文件 | 运行时直接读 classpath 文件，不用 DB | 零运维，git 版本控制自带审计 | 修改要重启，无管理界面 |
| C. 文件优先 | 文件是主源，启动时自动同步到 DB | 开发体验好（改文件即生效），DB 只做缓存层 | 文件改了 DB 自动更新？还是人工触发？容易两边不一致 |
| D. Git + Webhook | 提示词放独立 git 仓库，push 触发 webhook → 拉取 → reload | 完整的 git 审计 + PR review 流程 | 重，适合提示词频繁变更且有协作需求的团队 |

**评价：** 当前方案 A 方向对，但在 seed 逻辑上不完整——文件改了再 seed 不生效（幂等逻辑只查 key 不查内容 hash）。建议补一个"文件变更时 seed 更新"的能力，或者在管理端明确标注"此提示词已从文件初始化，文件版本可能更新"。

---

### 决策 2：INSERT-only 版本策略

**当前做法：** 每次修改 INSERT 新记录（version + 1），旧记录 deactivate。审计日志记录 old_content → new_content 的完整 diff。

**为什么这样做：** 提示词修改频率可能很高且影响大，需要完整的回滚和追溯能力。比起"UPDATE + 单独审计表"，INSERT-only 天然带版本链。

**替代方案：**

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. INSERT-only（当前） | 每次新版一条记录 | 版本链自然，回滚简单（切 is_active） | 表行数增长快，去重靠人工 |
| B. UPDATE + 审计表 | 主表只存当前版本，审计表存历史 | 主表干净，查询快 | 审计表可能漏写，回滚要从审计表还原 |
| C. Git 式存储 | content 存完整文本，version 链存 diff | 节省存储，能看到每次改动点 | 实现复杂（文本 diff + patch），回滚需要 merge |

**评价：** 对于提示词这种文本量不大（每条几千到一万字符）且修改频率不高（不会每秒钟改）的场景，INSERT-only 是最简单的正确选择。不需要换成 C——文本 diff 的复杂度不值得。

---

### 决策 3：Caffeine 手动失效 vs 自动过期

**当前做法：** TTL=0（不过期），手动调 reload 接口刷新。

**为什么这样做：** 提示词不是"每隔一段时间自动刷新"，而是"我改完了通知你刷新"。手动失效避免无意义的定时查库，保证生效时间点的确定性。

**替代方案：**

| 方案 | 做法 | 何时生效 | 适用场景 |
|------|------|---------|---------|
| A. 手动失效（当前） | TTL=0，调 reload 刷新 | 人工操作后立即 | 提示词变更频率低、需人为确认 |
| B. 短 TTL（30s-5min） | 自动过期，下次查询时重新加载 | 最多等一个 TTL 窗口 | 提示词频繁改动，不想手动 reload |
| C. DB 变更触发 | 在 AiPromptMapper.update 中发 ApplicationEvent，缓存层监听并刷新 | DB 写入后秒级自动 | 改动频繁 + 不想等 + 不想手动 |
| D. 基于版本号 | 缓存存 (key, version, content)，查询时先查 version，DB version 变了才 reload | 零额外调用 | 性能最优但需要改 SQL |

**评价：** 当前只有 draw 一个模块时 A 完全够用。但如果未来多个模块多人同时管理提示词，建议升级到 C——在 `AiPromptServiceImpl.updatePrompt()` 中发布 `PromptChangedEvent`，缓存层监听自动刷新，不再需要人记着调 reload。

---

### 决策 4：Caffeine 缓存按全量 key 缓存 vs 按 module 分区

**当前做法：** 一个 `Cache<String, String>`，key 是 `"module:promptKey"`。reload 全量清空。

**问题：** 多模块时，draw 的变更导致 chat 的缓存也被清空，虽然 cache miss 后会重新加载，但这是不必要的数据库查询。

**替代方案：**

| 方案 | 做法 | reload 粒度 |
|------|------|------------|
| A. 全量清空（当前） | 一个 Cache，reload 全清 | 全量 |
| B. 按 module 分区 | `Map<String, Cache<String, String>>`，外层 key=module | 按 module reload |
| C. 单个 key 刷新 | reload 接受 module+key 参数，只清一个 entry | 单个 key |

**评价：** 当前只有 draw 一个模块，A 没问题。如果模块数超过 2 个，建议改 B——改动不大（AiPromptService 里 Map 替换 Cache），但避免了模块间的缓存干扰。

---

### 决策 5：模型路由策略

**当前做法：** PromptManager 中用字符串匹配判断模型能力：

```java
if (modelId.contains("claude-opus-4-5")) {
    extended = promptService.getPrompt("draw", "system-extended");
}
```

**为什么这样做不好：** 提示词内容已经外置了，但"什么模型用哪套提示词"还在代码里，每加一个新模型要改 Java。

**替代方案：**

| 方案 | 做法 | 加模型时 |
|------|------|---------|
| A. 代码 if-else（当前） | PromptManager 中字符串匹配 | 改代码 + 编译 + 部署 |
| B. site_config 映射 | `site_config` 表：key=`model.prompt.override.deepseek-v4-pro`，value=`system-extended` | 改配置即可，即时生效 |
| C. ai_prompts 元数据 | 在 `ai_prompts` 表加 `model_filter` 字段，PromptManager 查询时传入 modelId | 管理端打钩配置 |
| D. 提示词内条件语法 | 提示词支持 `{{#if model.isPowerful}}...{{/if}}` 模板语法 | 改提示词文本即可，最灵活但也最复杂 |

**评价：** B 是最合适的中间方案——改动最小（加一个 site_config key），不需要改表结构，不需要引入模板引擎。当 site_config 中无覆盖时走默认逻辑（用 system-default），有覆盖时用指定版本。

---

### 决策 6：灰度字段的设计

**当前做法：** `ai_prompts` 表有 `effective_from` / `effective_to` 字段，但 AiPromptService 查询只用 `is_active=1`，完全忽略时间。

**为什么会有：** 设计阶段超前考虑了灰度发布，但实现阶段发现没有实际需求，搁置了。

**替代方案：**

| 方案 | 做法 | 复杂度 |
|------|------|--------|
| A. 删除字段（推荐） | 删掉 effective_from/to，等真有需求时再加 | 零 |
| B. 实现定时生效 | `WHERE is_active=1 AND (effective_from IS NULL OR NOW() >= effective_from) AND (effective_to IS NULL OR NOW() < effective_to)` | 一行 SQL |
| C. 定时任务 + 自动激活 | 加 @Scheduled 每分钟扫描，到期自动切换 is_active | 一个定时方法 |

**评价：** 如果未来有"晚上 10 点上线下周活动的提示词"这种场景，B 的一行 SQL 就够。如果没有，A——删掉比留着误导好。

---

### 决策 7：Seed 的幂等语义

**当前做法：** `POST /api/v1/admin/ai/prompts/seed` 逻辑是"DB 已有 (module, prompt_key) 就不插入"。文件改动了再 seed 不会更新 DB。

**为什么这样做：** 防止 seed 意外覆盖生产环境中人工修改过的提示词。

**替代方案：**

| 方案 | 做法 | 安全性 | 便利性 |
|------|------|--------|--------|
| A. 永不复写（当前） | 存在就不插 | 最高，不会丢失人工修改 | 文件改了要手动在管理端同步 |
| B. 基于 checksum | seed 时比较文件 checksum 和初始 seed 记录的 checksum。如果文件变了且 DB 没被人工改过（checksum 仍匹配 seed 时的值），自动更新 | 中等，能区分"人改了 DB"和"文件改了" | 好 |
| C. force 参数 | `seed?module=draw&force=true` 强制覆盖 | 低，可能误操作覆盖 | 最高 |
| D. 仅校验+提醒 | seed 时比较内容，不一致时打 WARN 日志 + 返回 diff 列表 | 最高（不写） | 中（改不改由人决定） |

**评价：** 推荐 B + D 的组合：seed 时计算文件 checksum 和 DB 内容对比，如果 DB 未被人工修改过（和上次 seed 的 checksum 一致）则自动更新，如果被人改过则返回 diff 提醒。

---

### 决策 8：提示词内容校验

**当前做法：** 管理端保存提示词无校验，接受任意文本。

**问题：** 如果保存了残缺提示词（比如 LLM 需要的 JSON 模板少了大括号），下次 LLM 调用会直接失败。而且失败的根因排查困难——用户看到的是"AI 调用失败"，而不是"提示词格式错误"。

**替代方案：**

| 方案 | 做法 | 成本 |
|------|------|------|
| A. 保存时不做校验（当前） | — | 零，但风险在上面 |
| B. 静态规则校验 | 检查占位符配对（`{{` 和 `}}` 数量相等）、Markdown 基本结构完整 | 低，几十行正则 |
| C. 沙箱验证（推荐） | 保存时用最小的模型（如 deepseek-chat，temperature=0, max_tokens=1）发一个空查询，验证不抛异常 | 中，一次 API 调用（约半秒） |
| D. A/B 预览 | 保存后默认不生效，在管理端点"预览"用真实请求跑一遍看效果，确认后点"发布" | 中高，需要管理端 UI 配合 |

**评价：** B 是立刻可以加的（防御低级错误），C 是上线前值得加的（防止语义问题）。D 是理想态——但不急。

---

### 决策 9：路径安全

**当前做法：** `FilePromptLoader` 用 `module` 和 `key` 拼接 `classpath:prompts/{module}/{key}.md`，无字符校验。

**替代方案：**

| 方案 | 做法 |
|------|------|
| A. 无校验（当前） | 信任调用方传入合法值 |
| B. 白名单正则 | module/key 只允许 `[a-z0-9-]+`，不匹配则拒绝 |
| C. 枚举白名单 | module 只能是已注册的枚举值，key 从文件列表读取 |

**评价：** B 方案一行正则，零依赖，零性能开销，防御了路径遍历。建议立刻加。

## 企业级方案对比

当前方案（DB + 文件兜底 + Caffeine）是轻量级的正确选择。但如果团队规模、业务复杂度、合规要求提升，需要考虑更企业级的方案。下面按"配置如何管理"这个核心问题展开。

### 维度一：配置中心的引入

Nacos、Apollo、Consul 这些是"配置中心"——当项目有几十个微服务、每个服务有几百个配置项时，不可能每个服务各自维护一份 application.yml。配置中心解决的是"所有服务的所有配置放在一个地方统一管理"。

**它们和提示词管理的关系是什么？** 提示词本质上也是一种配置——只不过它的值是一万字符的 Markdown 而不是 `timeout=30`。所以理论上可以用配置中心管提示词，但实际上大多数配置中心不是为"长文本频繁修改"设计的。

| 产品 | 定位 | 长文本支持 | 版本审计 | 灰度发布 | 运维成本 |
|------|------|----------|---------|---------|---------|
| **Nacos** | 阿里开源，服务发现 + 配置管理 | 有长度限制（默认 100KB），非设计目标 | 有历史版本，但 diff 展示弱 | 支持标签灰度 | 需部署 Nacos Server（Java 进程 + MySQL） |
| **Apollo** | 携程开源，专注配置管理 | 同上，非设计目标 | 有发布历史和回滚 | 支持灰度发布（很成熟） | 需部署 Portal + Admin + Config Service（较重） |
| **Consul** | HashiCorp，服务发现 + KV 存储 | KV 值可存长文本，但无 diff/审计 | 无内置版本审计 | 无内置灰度 | 单二进制部署（轻） |
| **Spring Cloud Config** | Spring 官方，Git-backed 配置 | 没有限制（本来就是文件） | Git 自带完整审计 | 靠 Git 分支管理 | 轻（Git 仓库 + 一个 Config Server） |
| **自建 DB + API（当前方案）** | 自己管 | 无限 | 自带了审计表 | 预留了字段未实现 | 轻（已有 MySQL） |

下面逐个分析在你的场景下的适用性。

---

### 方案 A：Nacos

**架构：**

```
Nacos Server
  ├── namespace: lblog
  │   ├── group: AI_PROMPTS
  │   │   ├── dataId: draw.system-default  → content: "你是一个..."
  │   │   ├── dataId: draw.system-extended → content: "额外规则..."
  │   │   └── dataId: draw.style-minimal   → content: "极简风格..."
  │   └── group: AI_CONFIG
  │       ├── dataId: model.mapping        → content: {"deepseek-v4-pro": "system-extended"}
  │       └── dataId: compression.window   → content: "20"

lblog-server
  └── AiPromptService 改为调用 Nacos SDK
       └── configService.getConfig("draw.system-default", "AI_PROMPTS")
```

**优点：**
- 配置变更秒级推送（长轮询），不需要手动 reload
- 管理端自带（Nacos Console），不用自己写 AdminPromptController 了
- 支持 namespace 隔离环境（dev/staging/prod）
- 阿里开源、中文社区活跃、Spring Cloud Alibaba 自动集成

**缺点：**
- 需要额外部署一个 Nacos Server（一个 Java 进程 + 它自己的 MySQL），运维成本增加
- 提示词是长文本，Nacos 管理端 web UI 编辑长文本体验一般（没有 Markdown 预览、没有 diff 高亮）
- 审计能力较弱——只记录"谁改了 key"，没有 old_content → new_content 的完整 diff
- 版本回滚要依赖 Nacos 的历史版本功能，和你的 INSERT-only 方案比各有利弊
- **引入了一个外部服务依赖。Nacos 挂了 = AI 提示词加载失败。** 虽然有 SDK 本地快照兜底，但故障恢复链路变长了

**适用判断：** 如果你已经有 Nacos 在用（或计划引入微服务架构），提示词挂上去是顺手的事。如果只有 lblog 这一个服务在用，为提示词单独部署 Nacos 是杀鸡用牛刀。

---

### 方案 B：Apollo

**架构：**

```
Apollo Portal (管理界面)
    ↓
Apollo Admin Service (写)
    ↓
Apollo Config Service (读，集群)
    ↓ MySQL
    ↓
lblog-server ← 长轮询拉配置
```

**优点：**
- **灰度发布是 Apollo 的核心竞争力。** 可以做到"5% 用户用新版提示词，95% 用旧版"，跑几天看效果再全量推开。这对提示词优化非常有用
- 版本 diff 和回滚比 Nacos 好（管理端直接对比两个版本的内容）
- 权限模型更细（按应用 + namespace 授权）
- 携程在生产环境大规模验证（万级应用、百万级配置项）

**缺点：**
- **部署复杂度是三者中最高的。** 需要 Portal + Admin Service + Config Service + Meta Server，再加 MySQL。即使是单机部署，运维负担远大于 Nacos 和 Consul
- 长文本编辑体验同样一般（它也是为 Key-Value 配置设计的）
- 学习曲线较陡（概念多：AppId、Namespace、Cluster、GrayRelease Rule...）
- 社区活跃度近年下降（携程内部仍在用，但开源社区更新速度放缓了）

**适用判断：** 如果你的提示词需要**精细的灰度策略**（按用户百分比、按 IP 段、按标签分组），Apollo 是三个配置中心里最强的。但为这一个需求引入 Apollo 的部署复杂度，只适合已经有运维团队的中大团队。

---

### 方案 C：Consul KV

**架构：**

```
Consul Agent
  └── KV Store
       ├── lblog/ai/prompts/draw/system-default → "你是一个..."
       └── lblog/ai/prompts/draw/system-extended → "额外规则..."

lblog-server
  └── Spring Cloud Consul Config
       └── @RefreshScope 自动热更新
```

**优点：**
- **部署最轻。** 一个二进制文件 `consul agent -dev` 就起来了，不需要 MySQL
- KV 天然支持任意长度的 value
- 自带健康检查（Consul 的老本行）
- Spring Cloud Consul 自动集成，`@RefreshScope` 可以让 prompt 自动热更新

**缺点：**
- **没有版本审计。** KV store 不记录历史版本，谁改了、改了什么完全不知道
- 没有灰度发布
- 管理界面比 Nacos/Apollo 简陋（本质是 KV 浏览器，不是配置管理平台）
- 长轮询不是 Consul 的优势场景（它更擅长服务发现和健康检查）

**适用判断：** 如果你的团队已经用了 Consul 做服务发现，顺手用它存提示词是可以的。但如果你是因为"想管提示词"而引入 Consul，不推荐——它缺乏审计和版本能力，这两个恰恰是提示词管理的核心需求。

---

### 方案 D：Spring Cloud Config + Git

**架构：**

```
Git 仓库 (prompts-repo)
  ├── prompts/
  │   ├── draw/
  │   │   ├── system-default.md
  │   │   └── system-extended.md
  │   └── chat/
  │       └── system-default.md
  └── model-mapping.yml  → {deepseek-v4-pro: system-extended}

Spring Cloud Config Server
  ↓ 拉取 Git 仓库
lblog-server
  ↓ 从 Config Server 拉配置
```

**优点：**
- **Git 是全世界最好的版本管理系统。** PR review、diff、blame、cherry-pick、rollback 全部免费获得
- 提示词作者可以用 IDE 编辑 Markdown（有预览、有拼写检查、有 linter）
- 不需要写 AdminPromptController、不需要建 `ai_prompts` 表、不需要审计表——Git 全包了
- Spring Cloud Config Server 可选不要（直接用 Spring Cloud Config Client 连 Git），零额外服务
- 可以通过 GitLab/GitHub 的 CODEOWNERS 做审批门禁——提示词变更必须通过 PR 且至少一个人 approve

**缺点：**
- **热更新路径更长。** 改提示词 → push → (Code Review) → merge → Config Server reload → 服务 `@RefreshScope` 刷新。不像 DB 方案秒级生效
- 需要 Git 仓库权限管理（谁可以改提示词 = 谁有仓库写权限），权限粒度不如自建管理端灵活
- 如果有非开发角色管理提示词（运营、产品），让他们用 Git 有学习成本
- `@RefreshScope` 在某些 Spring 版本有 bug（Bean 重建时序问题）

**适用判断：** 提示词的编辑者是你或团队的另一个开发者——Git + classpath 文件可能是最自然的选择。你已经在用这种模式了（`resources/prompts/draw/*.md`），只是又多加了 DB 层。

---

### 方案 E：独立提示词管理平台

如果你对提示词管理的需求已经超出了"存一段文本"的范畴，需要以下能力：

- 提示词版本对比（并排 visual diff）
- 批量生成测试样本 → 自动评分
- 提示词 A/B 实验 → 统计胜出
- 多人协作编辑（类似 Google Docs）
- LLM 调用日志关联到具体版本（"这次回答用的是 version 12 的提示词"）

那自建 DB + CRUD 迟早会碰到天花板。以下是目前行业里的一些选项：

| 平台 | 定位 | 定价 | 适合 |
|------|------|------|------|
| **[LangSmith](https://smith.langchain.com)** | LangChain 官方的 LLM 可观测 + 提示词管理平台 | 免费额度 + 付费 | 已用 LangChain 的团队 |
| **[Agenta](https://github.com/Agenta-AI/agenta)** | 开源 LLM 工程平台（提示词管理 + 评测 + 观测） | 开源 | 想自建但不想从零写 |
| **[PromptLayer](https://promptlayer.com)** | 提示词版本管理 + 请求日志 + 评分 | SaaS 付费 | 需要开箱即用的 SaaS |
| **[Helicone](https://helicone.ai)** | LLM 可观测平台（侧重日志 + 成本 + 延迟） | 开源 / SaaS | 主要需求是观测 |

这些平台解决的问题比"存一段提示词"大得多——它们管的是**整个 LLM 调用的生命周期**（请求→响应→评估→迭代）。如果项目未来要走 P6（评估与反馈）路线，可以考虑直接上这类平台而不是自己搭。

---

### 怎么选：决策矩阵

| 你的需求 | 推荐方案 | 理由 |
|---------|---------|------|
| 一个服务，一个人管提示词，不需要灰度 | **当前方案（DB + 文件）** 就够 | 复杂度匹配，别过度设计 |
| 一个服务，但需要灰度发布提示词 | 当前方案 + 实现 `effective_from/to` 或引入 **Apollo** | 自建灰度要写定时任务，Apollo 自带 |
| 多服务共享同一套提示词 | **Nacos** 或 **Consul KV** | 一个地方改，所有服务同时生效 |
| 提示词编辑者是开发者，用 git 工作流 | **Spring Cloud Config + Git** | PR review + IDE 编辑 Markdown + 零额外服务 |
| 提示词频繁变更 + 需要 A/B 测试 | **LangSmith** 或 **Agenta** | 自建评测系统成本远高于接入平台 |
| 团队有非开发人员管提示词 | **当前方案** + 管理端 UI 优化 | 配置中心的管理界面对非开发不友好 |
| 项目未来规划了 P6（评估与反馈） | 先用当前方案，P6 前评估 **LangSmith/Agenta** | 提前埋点（请求日志带 prompt_version），切换时数据不丢 |

---

### 整体评价

当前的 DB + 文件方案对于一个服务、一个开发者、一个 Agent 的场景是**架构复杂度刚好**的选择。它不是"不够企业级"——它是"不需要企业级"。

真正应该警惕的不是"没有用 Nacos/Apollo"，而是当前方案里几个**无论用什么基础设施都该做对的事**：
1. 模型路由从代码移到配置（不管是 DB 还是 git 文件）
2. 内容保存前做基础校验（静态规则就够了）
3. 路径安全加白名单
4. 缓存按 module 隔离

这四件事修了之后，如果未来需要升级到 Nacos/Apollo/Config Server，迁移代价很低——因为 `AiPromptService` 接口后面换什么存储实现都只要改一个类。
