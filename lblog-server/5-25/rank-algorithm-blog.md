# 别再按点赞数排序了——内容推荐算法的正确打开方式

一个内容网站总有几个排序 Tab：「推荐」「最新」「最热」。新手做法是：

- 推荐 → `ORDER BY like_count DESC`
- 最热 → `ORDER BY view_count DESC`
- 最新 → `ORDER BY published_at DESC`

看起来没问题。上线三个月后你会发现，推荐和最热是同一批文章，而且永远不会变。

## 问题在哪

点赞数和浏览量都是**累加值**。一篇三年前的文章积累了 100 个赞，昨天发布的一篇好文章只有 20 个赞——纯按点赞排序，新内容永远翻不了身。

排行榜变成了化石。这不是排序，这是考古。

真正的问题是：**没有时间维度**。

## Hacker News 怎么做

Hacker News 的排名公式大概是业界最知名的：

```
Score = (upvotes - downvotes) / (hours_since_post + 2)^1.5
```

分子的赞踩差代表"质量"，分母的时间衰减代表"时效"。指数 1.5 意味着时间越久分数跌得越快——不是线性的，是加速下沉。

一篇文章发布 1 小时得 10 票，另一篇发布 10 小时得 10 票，前者的得分远高于后者。这不是 bug，是 feature——首页应该展示"此刻"的热门，不是"历史上"的热门。

## Reddit 的做法

Reddit 的热门排序用了不同的数学工具：

```
Score = log10(|ups - downs|) × sign + published_timestamp / 45000
```

前半部分是置信度（对数压缩让 1 票和 100 票的差距不那么大），后半部分是时间（越新的时间戳越大）。两者相加，质量和时效各占一半权重。

Reddit 还有一个著名的"最佳"排序，基于 Wilson 置信区间——不过那是另一个话题了。

## 通用公式：时间衰减加权评分

把 HN 和 Reddit 的思路抽象一下，核心公式可以写成：

```
Score = Σ(signal × weight) / (age + base)^decay
```

### 信号（Signal）

互动数据。但不同互动的"含金量"不同：

- **评论** — 权重最高。写评论比点个赞成本大得多，说明内容触动了读者
- **点赞** — 次之。是认可但不一定有深度互动
- **浏览** — 最低。可能点进来就关了

一篇有 3 条评论的文章，互动深度远超过一篇 50 赞但没人说话的文章。

### 时间衰减（Decay）

- `age` — 内容发布多少天了
- `base` — 起飞窗口。`age + base` 让新文章有缓冲期，不会因为 age=0 而爆炸
- `decay` — 重力指数。1.2 是温和衰减（两周内仍有竞争力），1.5 是激进衰减（七天外基本沉底），2.0 以上是"只看昨天"

**base 怎么选？**

- `base=1`：昨天的文章就已经跌了一半
- `base=2`：前 2 天是起飞窗口，文章有机会冲榜
- `base=7`：前一周都有竞争力，适合更新慢的站点

**decay 怎么选？**

| decay | 效果 |
|-------|------|
| 1.0 | 线性衰减，新旧公平竞争 |
| 1.2 | 温和，两周内仍有竞争力 |
| 1.5 | 标准 HN 参数，几天后快速下沉 |
| 1.8 | 激进，只看近期 |
| 2.0+ | 今日热榜模式 |

### 一个具体例子

假设两篇文章：

| | 老文 A | 新文 B |
|---|--------|--------|
| 点赞 | 100 | 20 |
| 评论 | 10 | 5 |
| 浏览 | 5000 | 200 |
| 年龄 | 1095 天 | 5 天 |

**纯点赞排序**：老文 A 排第一，新文 B 排第二。分值差 5 倍。

**加权时间衰减**（推荐模式：like×2 + comment×3 + view×0.05，base=2，decay=1.2）：

- 老文 A：`(100×2 + 10×3 + 5000×0.05) / (1095+2)^1.2` = `480 / 4296` ≈ **0.11**
- 新文 B：`(20×2 + 5×3 + 200×0.05) / (5+2)^1.2` = `65 / 10.3` ≈ **6.31**

新文 B 的得分是老文 A 的 **57 倍**。这才是合理的——5 天就能拿到 20 赞 5 评论的文章，显然比三年前的老文有更高的"当前价值"。

## 优点

**1. 排行榜永远是活的**

新内容有机会浮上来，老内容自然下沉。用户每次刷新都能看到变化，而不会觉得"这个站好像很久没更新了"。

**2. 参数可解释**

不像推荐算法黑盒（"算法给你推了这个"），时间衰减公式的每个参数都有物理意义——weight 是什么信号的权重，decay 是重力大小。运营人员可以理解，可以调试。

**3. 不依赖外部数据**

纯 SQL 计算，不需要用户画像、行为日志、协同过滤。中小站点最怕引入推荐系统——数据不够、维护成本高、效果还不好。用这个公式在 ORDER BY 里算一下就行。

**4. 抗刷**

纯点赞排序很容易被刷——找几个人点赞就能霸榜。时间衰减天然限制了这个操作窗口：你只能在发布后的"起飞窗口"内操作，过了这个窗口刷再多赞也顶不上来。

## 缺点

**1. 计算列不能走索引**

分数是实时计算的，MySQL 必须 filesort。每次查询都要对符合条件的文章逐行计算评分。不过对于内容量 < 5000 的站点，这完全不是问题——WHERE 条件先过滤，剩下几百行，算一下毫秒级完成。

**2. 零互动文章全部排底**

一篇刚发布的高质量长文，如果还没人互动，评分就是 0。需要二级排序兜底（比如按发布时间降序），或者考虑给新文章一个"冷启动权重"。

**3. 调参是玄学**

`base=2` 还是 `base=3`？`decay=1.2` 还是 `1.3`？没有绝对正确的答案。不同站点的内容更新频率、用户互动习惯不同，最优参数也不同。建议做成可配置的，先上默认值，观察一两周数据再微调。

**4. 不适合超大规模**

如果文章量到 10 万级，这个方案就不够了——每次查询都 filesort 十万行不现实。这个量级需要预计算评分（定时任务写入数据库字段加索引），或者把排序交给搜索引擎（Elasticsearch 的 function_score）。

## 什么时候用

| 场景 | 适合？ |
|------|--------|
| 个人博客 / 小型社区 | 极其适合，简单有效 |
| 中型内容站（几千到几万篇） | 适合，filesort 仍可接受 |
| 大型平台（百万级） | 需要预计算或 ES |
| 实时热点（微博/Twitter） | 不够，需要实时流计算 |
| 个性化推荐 | 不适合，这不是推荐算法 |

## 本博客的实践

说完理论，聊聊本博客是怎么落地的。四种排序，各有分工：

### 推荐

目标和「编辑精选」不同——推荐不是把点赞最多的文章钉在首页，而是让**好内容持续有机会被看见**。

```
Score = (like×2 + comment×3 + view×0.05) / (age + 2)^1.2
```

评论权重最高（3），一篇有深度讨论的老文章仍然有价值，但不会压制新内容。decay 选了 1.2，比较温和，两周内的文章都有机会上榜。base 选 2，给新文章 48 小时的"起飞窗口"。

### 最热

偏短期热度，强调"现在什么最火"。

```
Score = (view×0.1 + like×1 + comment×2) / (age + 1)^1.5
```

浏览权重拉高（0.1），decay 选 1.5——和 Hacker News 一样，七天后热度快速消退。base 只有 1，昨天的文章今天就面临"毕业"。用户打开最热 tab，看到的一定是近期活跃的内容。

### 最新

不做任何加权，纯粹的倒序排列。不是所有东西都需要「算法」，时间线本身就是最好的排序。

### 专栏顺序

按作者手动指定的顺序排列。系列文章有内在的阅读次第，机器不应替作者做主。

### 参数全部可配

上面那些 2.0、3.0、1.2、1.5 都不是写死的。每个权重和 decay 都是一个配置项，可以在后台动态调整。觉得最近被刷赞了？把点赞权重降一点。觉得老文章沉太快了？decay 从 1.5 调到 1.3。不用改代码，不用重启服务，后台改个数字，30 分钟内生效。

这也意味着算法不是在真空里设计好的——上线跑一周数据，观察各 tab 的排名分布，再决定要不要调。**先上默认值，再让数据说话。**

### 权衡

每篇文章查询时实时计算评分，走 MySQL 的 filesort。对于目前的文章量（几十到几百篇）完全可以忽略。未来如果文章量到数千篇级别，仍然在可接受范围内；万一到了数万篇，再考虑预计算评分的方案也不迟。

## 总结

时间衰减加权评分是一种**简单、可解释、低成本的排序方案**。它只需要一个公式和几个参数，就能把一个"死了"的排行榜变成"活的"。

核心就一句话：**好的内容值得被发现，但"好"不等于"老"。** 你的排序算法应该反映这个直觉。
