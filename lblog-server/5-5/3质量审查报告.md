## 逐项审查

### 任务 1: TokenGenerator

**发现的问题：**
1. **`hash()` 方法未指定字符集** -- `rawToken.getBytes()` 使用平台默认字符集，在不同操作系统上行为可能不一致。虽然后续 `generateRandomToken()` 只产生 Base64 URL-safe 的 ASCII 字符，但作为安全组件应显式指定字符集。
   - 修复：改为 `rawToken.getBytes(StandardCharsets.UTF_8)`

2. **`bytesToHex` 使用 `String.format` 逐字节转换** -- 循环内调用 `String.format("%02x", ...)` 每次创建新的 `Formatter` 对象，对频繁调用的工具方法而言效率较低。（优化级别：低优先级）

**优化建议（已执行）：**
- 已添加 `java.nio.charset.StandardCharsets` 导入，将 `getBytes()` 改为 `getBytes(StandardCharsets.UTF_8)`。

**设计合规检查：**
- `@Component` -- 符合
- `@Value` 注入两个超时属性 -- 符合（额外增加了 `token-byte-size` 配置，合理扩展）
- `generate()` 返回 `TokenPairRaw` -- 符合，字段与设计完全一致
- `hash(String)` 返回 SHA-256 -- 符合
- `TokenPairRaw` 定义为 `public static` 内部类 -- 符合，使用 `@Data` + `@Accessors(chain=true)` 合理

### TokenRecord

**发现的问题：**

1. **`createdAt` 字段不在设计文档中** -- 设计文档的 `TokenRecord` 类定义包含 7 个字段（不含 `createdAt`），但实现增加了 `createdAt`。该字段存在于 `user_tokens` 表并映射到 `UserToken` 实体，添加到 `TokenRecord` 对调用方有用（可判断 token 创建时间），属合理扩展。

2. **`revoked` 类型差异** -- 设计文档中 `revoked` 使用 `boolean`（原始类型），`UserToken` 实体中使用 `Boolean`（包装类）。`DbTokenRepository.toRecord()` 做了正确空安全转换：`entity.getRevoked() != null && entity.getRevoked()`。类型差异合理，领域对象(DTO)不应为 null。

### 任务 4: DbTokenRepository

**发现的问题（已修复）：**

1. **`countValidByUserId` 实现效率低** -- 原代码通过 `userTokenMapper.findValidByUserId(userId).size()` 查询，会加载所有匹配行的全部字段到内存再计数。对于有大量 token 的用户来说浪费网络传输和内存。
   - 修复：新增 `countValidByUserId` Mapper 方法和 SQL `SELECT COUNT(*)`，`DbTokenRepository` 直接调用。

**其他检查：**
- `@Repository` 注解 -- 正确
- 构造函数注入 -- 正确（Spring Boot 3.x 单构造函数自动注入，无需 `@Autowired`）
- `toEntity` / `toRecord` 转换 -- 正确，`revoked` 字段做空安全处理
- `save()` 方法将 MyBatis 生成的 ID 写回 `TokenRecord` -- 正确
- 所有委托方法均直接传递参数给 Mapper -- 正确
