## 二、BUG 修复记录

本次测试过程中发现并修复了 **4 个Bug**：

### Bug 1: SecurityConfig 公开接口 URL 模式错误

- **严重度**: 🔴 高
- **发现时间**: 代码审查阶段
- **描述**: `SecurityConfig.java` 中配置 `.requestMatchers(HttpMethod.GET, "/api/v1/home/**").permitAll()`，但 `HomeController` 的映射路径是 `@RequestMapping("/api/v1")` 而非 `/api/v1/home/`。导致所有公开接口（文章列表、分类、标签、专栏、点赞等）均需要认证，未登录无法访问前台页面。
- **修复**: 将 permitAll 模式改为实际端点路径：
  ```java
  .requestMatchers(HttpMethod.GET, "/api/v1/posts/**", "/api/v1/categories/**", "/api/v1/tags/**", "/api/v1/series/**").permitAll()
  .requestMatchers(HttpMethod.POST, "/api/v1/posts/*/view", "/api/v1/posts/*/like").permitAll()
  .requestMatchers(HttpMethod.DELETE, "/api/v1/posts/*/like").permitAll()
  .requestMatchers(HttpMethod.GET, "/api/v1/posts/*/like/status").permitAll()
  ```
- **涉及文件**: `SecurityConfig.java:62`

### Bug 2: @RequestHeader X-Visitor-Id 未设置 required=false

- **严重度**: 🟡 中
- **测试用例**: H-42, H-43
- **描述**: `likePost()` 和 `unlikePost()` 方法的 `@RequestHeader("X-Visitor-Id")` 默认 `required=true`。不传该 header 时 Spring 抛出 `MissingRequestHeaderException`，未被全局异常处理器捕获，返回 HTTP 500 而非预期的 400。
- **修复**: 添加 `required = false`，利用方法内已有的 null 检查返回 400。
- **涉及文件**: `HomeController.java:159,176`

### Bug 3: MaxUploadSizeExceededException 未处理

- **严重度**: 🟡 中
- **测试用例**: U-05
- **描述**: 上传超过 5MB 的文件时，Spring 的 multipart 限制在 Controller 之前就抛出 `MaxUploadSizeExceededException`，未被全局异常处理器捕获，返回 500 而非 400。
- **修复**: 在 `GlobalExceptionHandler` 中添加针对 `MaxUploadSizeExceededException` 的处理器。
- **涉及文件**: `GlobalExceptionHandler.java:46-50`

### Bug 4: UploadController @RequestParam 未设置 required=false

- **严重度**: 🟡 中
- **测试用例**: U-06
- **描述**: 不传 file 参数时，Spring 抛出 `MissingServletRequestPartException`，返回 500。
- **修复**: 添加 `required = false` + null 检查。
- **涉及文件**: `UploadController.java:36`

## 四、设计差异说明 (标为 FAIL 但非缺陷)

### 1. A-18: 重复登出返回 401

**测试计划预期**: code=0 (幂等)  
**实际行为**: code=401

**原因分析**: Token 吊销后，`TokenAuthenticationFilter` 验证失败 → 清除 SecurityContext → `ExceptionTranslationFilter` 检测到未认证 → 调用 `CustomAuthEntryPoint` 返回 401。这是 Token 认证架构的正常行为：吊销的 Token 不能访问任何端点。

**建议**: 这是设计选择，不是 Bug。测试计划应修改预期结果为 401。或者可以认为前端不需要重复调用登出。

### 2. A-22: 旧 Refresh Token 刷新仍返回 0

**测试计划预期**: code=401, 旧 token 已 revoked  
**实际行为**: code=0, 返回新 TokenPair

**原因分析**: `refreshAccessToken()` 实现了 replaced_by 链式机制。旧 token 被 revoked 但有 replaced_by 指向新 token，系统会沿着链查找有效 token 并签发新的。这是为多 Tab 并发场景设计的特性（A-23 验证了正常工作流程）。

**建议**: 测试计划的期望值需要根据实际设计调整。链式机制是特性非缺陷。如果需要严格的一次性 refresh（无链），需修改 `refreshAccessToken` 方法，在链式查找后也 revoke 替换 token。

### 3. H-01: 分页字段名为 `page` 非 `pageNum`

**测试计划预期**: 返回 `pageNum` 字段  
**实际行为**: 返回 `page` 字段

**原因**: `PageResult.java` 中的字段命名是 `page`。前端消费时需对应 `response.data.page`。这是字段命名约定问题，不影响功能。

## 六、总体评估

### 优点

1. **Token 安全机制成熟**: SHA-256 哈希存储、Token 类型分离（ACCESS/REFRESH）、吊销机制、Replaced-by 链式并发兜底、改密全量吊销
2. **分层架构清晰**: Controller → Service → Mapper，职责分明
3. **参数校验完善**: 大量使用 `@Valid` + `@Min/@Max` 注解
4. **公开接口安全**: SQL 注入防护 (MyBatis `#{}`)，越权访问防护 (`hasRole`)
5. **响应格式统一**: 全部采用 `ApiResponse<T>` 包装

### 修复的 Bugs

1. SecurityConfig 公开接口 URL 路径配置错误 (🔴高)
2. `@RequestHeader` 未设 `required=false` 导致 500 (🟡中)
3. `MaxUploadSizeExceededException` 未处理导致 500 (🟡中)
4. `@RequestParam` 未设 `required=false` 导致 500 (🟡中)

### 建议改进

1. **补充 `MissingServletRequestPartException` 等常见异常处理器**，避免框架层异常漏到 500
2. **统一测试计划与实现的预期值**：A-18 幂等登出、A-22 Refresh 链式机制
3. **H-01 `PageResult` 字段命名**：建议统一为 `pageNum` 或与前端对齐
4. **过期 Token 场景测试**：建议在 application.yml 中保留一组短期过期的测试配置
