### 6.3 Refresh 续期流程

| 步骤 | 描述 | 状态 |
|------|------|------|
| 1 | SHA-256 计算 hash | ✅ |
| 2 | findByHash 查找 | ✅ |
| 3 | 检查未过期、未吊销、type=REFRESH | ✅ |
| 4 | 吊销旧 refresh_token | ✅ (但未更新 replaced_by) |
| 5 | 签发新 TokenPair | ✅ |
| 6 | save() x 2 | ✅ |
| 7 | 多 tab replaced_by 兜底 | ✅ |

### 发现的问题

**问题 1: `findByHashForUpdate` 定义但未使用（高优先级）**

`TokenRepository` 和 `DbTokenRepository` 定义了 `findByHashForUpdate(String)` 方法，使用 `SELECT ... FOR UPDATE` 行锁。但 `TokenServiceImpl.refreshAccessToken()` 仅使用 `findByHash()` 无锁查询。在高并发刷新场景下，多个请求可能同时读取并认为 token 有效，导致竞争条件。

**问题 2: Refresh 未更新 `replaced_by` 字段（高优先级）**

设计文档描述：
> Rotation: 旧 refresh_token → revoked=1, replaced_by=新hash

但 `TokenServiceImpl.refreshAccessToken()` 仅执行 `tokenRepository.revoke(hash)` 吊销旧 token，**未设置 `replaced_by` 字段**指向新 token 的 hash。这意味着多 tab 并发时 `replaced_by` 链式查找逻辑虽然存在，但永远不会生效，因为 `replaced_by` 始终为 null。

理想实现：
```java
// 先更新 replaced_by，再吊销
userTokenMapper.updateReplacedBy(hash, pair.getRefreshHash());
tokenRepository.revoke(hash);
```

## 十、详细分项说明

### 10.1 密码编码器

SecurityConfig 中配置了 `DelegatingPasswordEncoder`，默认使用 BCrypt，匹配失败时回退到 NoOpPasswordEncoder（明文兼容）。 ✅

### 10.2 LoginUser 实现

`LoginUser.getAuthorities()` 返回 `ROLE_` + 角色名大写，与 SecurityConfig 的 `hasRole("ADMIN")` 配合正确。 ✅

### 10.3 TokenAuthenticationFilter

这是一个 `OncePerRequestFilter`，对所有请求进行 token 提取（包括 `/auth/login` 等公开端点）。不存在 token 时直接放行，由后续 SecurityConfig 的 `.anyRequest().authenticated()` 控制访问。 ✅

### 10.4 登出流程

取 `authentication.getCredentials()` 中存储的原始 token 字符串进行吊销。这依赖于 TokenAuthenticationFilter 在构建 `UsernamePasswordAuthenticationToken` 时将 token 作为 credentials 传入。 ✅

### 10.5 编译验证

项目通过 `JAVA_HOME="D:/ProgramFile/dev/java/jdk-17.0.16" mvn compile -DskipTests -q` 无错误编译。 ✅

## 十一、总体结论

### 已实现的核心安全能力

1. 完整的 Token 生成（SecureRandom + SHA-256）+ 配置驱动
2. TokenRepository 存储抽象层 + MySQL 实现，支持未来切换到 Redis
3. Token 验证过滤器链（TokenAuthenticationFilter）
4. 登录认证（AuthenticationManager + BCrypt）
5. 双 Token 签发（Access + Refresh）
6. Refresh Token Rotation + 多 tab 链式兜底
7. 路由权限矩阵（admin 端点需 ROLE_ADMIN）
8. AdminController 已去除硬编码 TEST_USER_ID
9. 401/403 JSON 错误响应

### 必须修复（高优先级）

1. **Refresh replaced_by 未设置** — 多 tab 并发处理形同虚设
2. **缺少 TokenCleanupTask** — 数据库持续膨胀
3. **缺少 change-password 端点** — 功能不完整

### 建议修复（中优先级）

4. Refresh rotation 缺少行级锁
5. 用户状态检查被跳过
6. 旧密码自动升级未实现
7. /error 未显式允许
