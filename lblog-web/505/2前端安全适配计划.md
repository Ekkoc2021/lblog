# 前端安全模块适配计划

> 基于后端安全模块（Access Token + Refresh Token 双令牌机制），对前端进行适配改造。

---

## 任务清单（共 6 项）

### Task 1：类型定义更新

**文件：** `src/types/index.ts`

新增 `TokenPairVO` 和 `RefreshRequest` 接口。`UserInfoVO` 可直接复用已有 `User` 接口（字段完全一致），无需重复定义。

```typescript
export interface TokenPairVO {
  accessToken: string;
  refreshToken: string;
  expiresIn: number;
  // User 接口已存在（id, username, nickname, avatar, email, role）
  user: User;
}

export interface RefreshRequest {
  refreshToken: string;
}
```

删除原 `LoginResponseData`（移到 api.ts 内部或删除）。

### Task 2：API 层重构（核心）

**文件：** `src/services/api.ts`

改动点：
1. **Token 读写** — `getToken()` 从 `lblog_token` 改为 `lblog_access_token`
2. **新增 `getRefreshToken()`** — 读取 `lblog_refresh_token`
3. **新增 `refreshToken()` 导出函数** — 调 `POST /api/v1/auth/refresh`
4. **401 拦截器改造** — 简单抛异常改为自动续期：
   - 引入 `isRefreshing` 锁 + 请求等待队列
   - Access token 过期 → 调 refresh → 更新本地双 token → 重放原请求
   - 刷新接口自身 401 不触发刷新（排除 `/auth/refresh`）
   - 多 tab 并发：只有一个发起 refresh，其余排队等待
   - ⚠️ **刷新失败时**：必须同时清除 `lblog_access_token` 和 `lblog_refresh_token`，并触发登出
5. **修改 `login()` 函数** — 返回类型适配 `TokenPairVO`，响应字段为 `accessToken` 非 `token`

---

### Task 3：AuthContext 重构

**文件：** `src/contexts/AuthContext.tsx`

改动点：
1. **双 token 存储** — 登录时存 `lblog_access_token` + `lblog_refresh_token`
2. **新增 `user` 状态** — `AuthContextType` 增加 `user: User | null`
3. **`updateTokens()` 方法** — 给 api.ts refresh 流程回调，更新 context 中的双 token（access + refresh）
4. **`login()` 改造** — 响应字段为 `accessToken`（非原 `token`），从 `TokenPairVO.user` 中提取用户信息保存
5. **`logout()` 调 API** — 先调用 `POST /api/v1/auth/logout`（后端吊销 token），再清本地存储

## 关键风险点

| 风险 | 说明 |
|------|------|
| 401 死循环 | refresh 接口自身 401 必须跳过刷新逻辑 |
| 竞态条件 | 多请求同时 401，确保只有一个发起 refresh |
| 请求重放 | 刷新成功后原请求 headers 需更新为新 token |
| Token 同步 | api.ts 和 AuthContext 中 token 需保持一致 |
| 刷新失败双清理 | refresh 失败时必须同时清除 access + refresh 两个 token，否则残留 refresh_token 导致前端状态不一致 |
